
# Part 1: 最終課題の要件

---

# 課題の全体像

### 自社データでGPTsを構築し、実運用して発表する

```
【最終課題で作るもの】

1. GPTs設計書
   └── 対象業務・要件・Instructions・知識ベース設計

2. 完成GPTs
   └── 動作するGPTs（ChatGPT Plus または API）

3. 運用ログ
   └── 実際の使用記録（5回以上）

4. 振り返りレポート
   └── フィードバック・改善記録

5. 発表資料
   └── 20分のプレゼンテーション
```

---

# 必須要件

| 項目 | 要件 | 補足 |
|---|---|---|
| **対象業務** | 自社・自部門の実業務 | 機密データは匿名化可 |
| **GPTs構成** | Instructions + 知識ベース（最低1ファイル） | Phase 3で学んだ手法を活用 |
| **実運用** | 自分で5回以上使用 OR 同僚1人に3回以上使用 | 実際の業務で使うこと |
| **フィードバック** | 使用ログ＋改善点を記録 | 改善サイクルを回す |
| **発表時間** | 20分（発表15分＋質疑5分） | 構成フォーマットあり |

---

# 対象業務の選び方

### 成功しやすい業務の特徴

| 特徴 | 理由 | 例 |
|---|---|---|
| **繰り返し発生** | 複数回使って改善できる | 週次レポート、定型問い合わせ対応 |
| **情報参照が多い** | 知識ベースが活きる | マニュアル参照、FAQ対応 |
| **フォーマット固定** | 出力品質が安定 | 報告書、議事録、メール |
| **自分で完結** | 依頼なしで使える | 自分の担当業務 |

---

# 避けたほうがよい業務

### 2週間では難しいケース

| 特徴 | 理由 |
|---|---|
| 年に1回しか発生しない | 実運用の機会がない |
| 高度な専門判断が必要 | GPTsだけでは対応困難 |
| 機密性が極めて高い | 匿名化しても使えない |
| 社内承認が複雑 | 2週間で承認が間に合わない |

> **ポイント**: 「小さく始めて確実に成果を出す」ことを優先

# Part 2: 2週間のスケジュール

---

# Week 12（設計・実装週）

```
【Week 12のスケジュール】

Day 1-2: 対象業務の選定、要件整理
    ├── Session 11の業務分析結果を活用
    ├── 対象業務を1つに絞る
    └── 必要な情報・データをリストアップ

Day 3-4: GPTs設計（メタプロンプト活用）
    ├── 企画書生成メタプロンプトで企画書作成
    ├── Instructions生成メタプロンプトでInstructions作成
    └── 知識ベース設計メタプロンプトで構造設計

Day 5-6: 知識ベース構築、GPTs実装
    ├── 知識ファイルを作成（匿名化含む）
    ├── GPTsを構築（ChatGPT Plus or API）
    └── 初期テスト実施

Day 7: 動作確認、初期テスト
    └── テストケースで動作検証
```

---

# Week 13（運用・振り返り週）

```
【Week 13のスケジュール】

Day 1-3: 実運用
    ├── 自分で5回以上使用 OR 同僚1人に3回以上使用
    ├── 使用ログを記録（日時・目的・結果・気づき）
    └── うまくいった点・課題を整理

Day 4-5: フィードバック収集、改善実施
    ├── ログを分析して改善点を特定
    ├── Instructions or 知識ベースを修正
    └── 再テストで効果確認

Day 6: 振り返り、発表資料作成
    ├── 振り返りレポート作成
    └── 発表スライド作成

Day 7: 最終発表（20分/人）
    └── 発表＋質疑応答
```

---

# Part 3: 発表フォーマット

---

# 最終発表の構成（20分）

### 発表15分＋質疑5分

| パート | 時間 | 内容 |
|---|:---:|---|
| 1. 対象業務と課題 | 3分 | どんな業務か、何が課題だったか |
| 2. GPTs設計・実装 | 5分 | 設計プロセス、Instructions・知識ベース概要、工夫点 |
| 3. 実運用の結果 | 5分 | 誰が・どう使ったか、対話ログ、効果・課題 |
| 4. 振り返りと今後 | 2分 | 学んだこと、今後の改善・展開計画 |
| 5. 質疑応答 | 5分 | 聴衆からの質問に回答 |

---

# パート1: 対象業務と課題（3分）

### 伝えるべき内容

```
【構成】

1. 業務の概要
   - 業務名、頻度、担当者
   - 具体的な作業内容

2. 課題・ペインポイント
   - 何が大変だったか
   - どれくらい時間がかかっていたか

3. GPTs化を選んだ理由
   - なぜこの業務を選んだか
   - 何を改善したかったか
```

---

# パート2: GPTs設計・実装（5分）

### 伝えるべき内容

```
【構成】

1. 設計プロセス
   - Session 11の業務分析結果をどう活用したか
   - メタプロンプトをどう活用したか

2. GPTsの概要
   - Instructions概要（ペルソナ、基本動作、出力形式）
   - 知識ベース構成（ファイル数、内容）

3. 工夫した点
   - 独自の工夫や改善
   - Phase 1-3の学びをどう活かしたか
```

---

# パート3: 実運用の結果（5分）

### 伝えるべき内容

```
【構成】

1. 運用方法
   - 誰が使ったか（自分 or 同僚）
   - どんな場面で使ったか

2. 対話ログ（抜粋）
   - 実際のやり取りを1-2件紹介
   - スクリーンショット or テキスト

3. 効果と課題
   - うまくいった点（時間削減、品質向上など）
   - まだ課題が残っている点
```

---

# パート4: 振り返りと今後（2分）

### 伝えるべき内容

```
【構成】

1. 学んだこと
   - 技術面の学び（メタプロンプト、GPTs設計など）
   - プロセス面の学び（業務分析、改善サイクルなど）

2. 今後の計画
   - さらに改善したい点
   - 他の業務への展開予定
   - チームへの展開予定
```

# 最終課題の評価基準

### 各20点 × 5観点 = 100点

| 観点 | 配点 | 評価基準 |
|---|:---:|---|
| **業務分析の質** | 20点 | 対象業務の課題が明確で、GPTs化の妥当性があるか |
| **GPTs設計・実装** | 20点 | メタプロンプトを活用し、適切な設計ができているか |
| **実運用の実績** | 20点 | 実際に使用し、ログが記録されているか |
| **改善サイクル** | 20点 | フィードバックに基づく改善を実施したか |
| **発表の質** | 20点 | 20分で成果と学びを効果的に伝えられたか |

---

# 評価の詳細（1/2）

### 業務分析の質（20点）

| 点数 | 基準 |
|:---:|---|
| 18-20 | 課題が具体的で定量化され、GPTs化の妥当性が高い |
| 14-17 | 課題が明確で、GPTs化の理由が説明されている |
| 10-13 | 課題はあるが、GPTs化との関連が弱い |
| 0-9 | 課題が不明確、または業務分析が不十分 |

### GPTs設計・実装（20点）

| 点数 | 基準 |
|:---:|---|
| 18-20 | メタプロンプト活用、Instructions・知識ベースが最適化されている |
| 14-17 | 適切な設計で、動作するGPTsが完成している |
| 10-13 | GPTsは完成しているが、設計に改善余地がある |
| 0-9 | 未完成、または動作しない |

---

# 評価の詳細（2/2）

### 実運用の実績（20点）

| 点数 | 基準 |
|:---:|---|
| 18-20 | 5回以上使用し、詳細なログが記録されている |
| 14-17 | 必要回数を使用し、ログが記録されている |
| 10-13 | 使用回数が不足、またはログが不十分 |
| 0-9 | 実運用していない、またはログがない |

### 改善サイクル（20点）

| 点数 | 基準 |
|:---:|---|
| 18-20 | 複数回の改善を実施し、効果を検証している |
| 14-17 | 1回以上の改善を実施し、記録がある |
| 10-13 | 改善を試みたが、効果検証が不十分 |
| 0-9 | 改善を実施していない |
