# Part 1: 使用ログの記録

---

# なぜ使用ログが重要か

### 「使った」だけでは改善できない

```
【ログなしの場合】
「何回か使った」「なんとなく便利だった」
→ 改善点が見えない、発表で具体性がない

【ログありの場合】
「5回使って、3回目から精度が上がった」
「〇〇の質問には弱いことがわかった」
→ 具体的な改善点が見える、発表に説得力がある
```

---

# 使用ログの記録項目

### 必須5項目

| 項目 | 内容 | 例 |
|---|---|---|
| **日時** | いつ使ったか | 2026/1/6 14:30 |
| **目的** | 何のために使ったか | 週次レポートのドラフト作成 |
| **入力** | 何を入力したか | 先週の売上データと課題 |
| **出力** | 何が返ってきたか | レポート案（800文字） |
| **評価** | どう評価するか | ○ / △ / × + コメント |

---

# ログ記録テンプレート

```
【使用ログ #1】

日時: 2026/1/6 14:30
目的: 週次レポートのドラフト作成
入力:
  - 先週の売上: 150万円（前週比+10%）
  - 課題: 新規顧客獲得が鈍化
出力:
  - 800文字のレポート案
  - 3つの改善提案付き
評価: ○
コメント:
  - 数値の要約が正確
  - 改善提案が具体的
  - 次回は「競合分析」も追加したい
```

---

# 評価基準（○△×）

### シンプルに3段階で評価

| 評価 | 基準 | 次のアクション |
|:---:|---|---|
| **○** | 期待通り、または期待以上 | 同様の使い方を継続 |
| **△** | 部分的に使える、要修正 | 改善点を特定、次回で検証 |
| **×** | 使えない、やり直し | 根本的な改善が必要 |

> **ポイント**: △が最も重要。「どこがダメで、何を直せば○になるか」を記録

---

# ログ記録のコツ

### 「後で書く」は書けない

```
【失敗パターン】
・使い終わってから「あとでまとめて書こう」
  → 忘れる、曖昧になる

【成功パターン】
・使いながらメモを取る
・テンプレートをコピペして埋める
・スクリーンショットを撮っておく

【おすすめ】
・スプレッドシートでログ管理
・Notionやメモアプリでリアルタイム記録
```

# Part 2: フィードバック収集

---

# フィードバックの種類

### 自分で使う場合 vs 同僚に使ってもらう場合

| パターン | メリット | デメリット |
|---|---|---|
| **自分で使う** | すぐ試せる、詳細なログが取れる | 客観性に欠ける |
| **同僚に使ってもらう** | 客観的な評価が得られる | 調整が必要、ログが曖昧になりがち |

> **推奨**: まず自分で使い、ある程度安定したら同僚に依頼

---

# 同僚に使ってもらう場合

### 事前準備が成功のカギ

```
【依頼時に伝えること】

1. GPTsの目的
   「〇〇業務を効率化するために作りました」

2. 使い方（簡単な説明）
   「〇〇と入力すると、△△が返ってきます」

3. フィードバックのお願い
   「使ってみて気づいたことを教えてください」
   「3回以上使っていただけると嬉しいです」

4. ログ記録のお願い
   「このシートに記録をお願いします」
```

---

# フィードバック収集テンプレート

### 同僚用の簡易フォーム

```
【GPTs使用フィードバック】

名前: _____________
使用日: ___/___/___
使用回数: ___回

1. 良かった点（複数可）
   □ 時間が短縮できた
   □ 品質が向上した
   □ 操作が簡単だった
   □ その他: _____________

2. 改善してほしい点（複数可）
   □ 出力が期待と違った
   □ 操作がわかりにくい
   □ 反応が遅い
   □ その他: _____________

3. 自由コメント
   ___________________________
```

---

# フィードバックの質を上げる質問

### 具体的に聞く

| 曖昧な質問 | 具体的な質問 |
|---|---|
| 「どうでしたか？」 | 「期待した出力と実際の出力は一致しましたか？」 |
| 「使いやすかったですか？」 | 「入力に迷った点はありましたか？」 |
| 「改善点はありますか？」 | 「〇〇の部分は、どう変えると使いやすいですか？」 |

# Part 3: 改善サイクル

---

# 改善サイクル（PDCA）

### 1回の改善で終わらない

```
【PDCAサイクル】

Plan（計画）
└── 改善点を特定、修正方針を決める

  ↓

Do（実行）
└── Instructions or 知識ベースを修正

  ↓

Check（確認）
└── 修正後のGPTsをテスト

  ↓

Act（改善）
└── 効果を評価、次の改善点を特定

  ↓
（繰り返し）
```

---

# 改善の優先順位

### 影響度×実現可能性で判断

| 優先度 | 影響度 | 実現可能性 | 例 |
|:---:|:---:|:---:|---|
| **高** | 高 | 高 | 出力形式の修正 |
| **中** | 高 | 低 | 大規模な知識ベース追加 |
| **中** | 低 | 高 | 細かいトーン調整 |
| **低** | 低 | 低 | 全面的な再設計 |

> **2週間の制約**: 優先度「高」に集中、「低」は今後の課題に

---

# よくある改善パターン

### Instructions編

| 症状 | 原因 | 改善方法 |
|---|---|---|
| 出力が長すぎる | 長さ制約がない | 「〇〇文字以内」を追加 |
| トーンが合わない | ペルソナ設定が曖昧 | 具体的な話し方を指定 |
| 質問に答えられない | 対応範囲が狭い | 「〇〇の質問には△△と回答」を追加 |

---

# よくある改善パターン

### 知識ベース編

| 症状 | 原因 | 改善方法 |
|---|---|---|
| 情報が古い | 知識ベースが更新されていない | 最新情報を追加 |
| 検索精度が低い | チャンク設計が不適切 | 見出しを追加、構造を改善 |
| 情報が見つからない | 知識ベースに含まれていない | 新しいファイルを追加 |

---

# 改善ログテンプレート

```
【改善ログ #1】

改善日: 2026/1/8
改善前の問題: 出力が長すぎて読みにくい

改善内容:
  - Instructionsに「出力は300文字以内」を追加
  - 「箇条書きで簡潔に」を追加

改善後の結果:
  - 出力が300文字程度に短縮
  - 箇条書きで読みやすくなった

効果: ○（期待通り）
次の改善点: 数値データの表形式出力を追加したい
```

# Part 4: 振り返りの構造化

---

# 振り返りの目的

### 「やった」を「学び」に変える

```
【振り返りなしの場合】
「GPTsを作って使った」
→ 何を学んだか不明確、発表が薄い

【振り返りありの場合】
「〇〇の工夫が効果的だった」
「△△は想定外の課題だった」
「次は□□を改善したい」
→ 具体的な学びが言語化される、発表に深みが出る
```

---

# KPT法（振り返りフレームワーク）

### シンプルで効果的な3分類

| 項目 | 意味 | 例 |
|---|---|---|
| **K**eep | 続けたいこと | メタプロンプトでInstructions生成 |
| **P**roblem | 問題だったこと | 知識ベースの検索精度が低い |
| **T**ry | 次に試したいこと | ファイル形式をJSONからMDに変更 |

---

# KPT振り返りテンプレート

```
【最終課題 振り返り（KPT）】

■ Keep（良かった、続けたい）
  - メタプロンプトで企画書からInstructionsを生成
    → 品質が安定、時間も短縮
  - 使用ログをスプレッドシートで管理
    → 改善点が見えやすかった

■ Problem（問題、課題）
  - 知識ベースの検索精度が低かった
    → 構造化が不十分だった
  - 同僚のフィードバック収集が遅れた
    → 事前説明が不足していた

■ Try（次に試したい）
  - 知識ベースをMarkdown + 見出し構造に変更
  - フィードバックシートを事前に共有
```

---

# 学びの整理（発表用）

### 技術面とプロセス面に分ける

```
【技術面の学び】

・メタプロンプトの効果
  - 「〇〇を入力すると△△を出力する」形式が汎用的
  - 企画書→Instructions→知識ベースの連鎖が効率的

・GPTs設計のポイント
  - 出力形式を明示すると品質が安定
  - 知識ベースは構造化が重要

【プロセス面の学び】

・業務分析の重要性
  - ボトルネック特定が正確だと、GPTs設計がスムーズ

・改善サイクルの価値
  - 1回目より3回目の方が精度が高い
  - ログがあると改善点が明確
```

# Part 5: 発表準備のコツ

---

# 発表で差がつくポイント

### 「やったこと」ではなく「学んだこと」

```
【弱い発表】
「GPTsを作りました」
「5回使いました」
「改善しました」
→ 事実の羅列、学びが見えない

【強い発表】
「〇〇の工夫をしたら△△が改善しました」
「想定外の課題は□□で、◇◇で解決しました」
「この経験から、次は▽▽を試したいです」
→ 学びと成長が見える、聴衆の参考になる
```

---

# 発表資料のチェックリスト

### 20分で伝えきれるか

```
【スライド構成チェック】

□ パート1（3分）: 業務と課題が1-2枚で説明できる
□ パート2（5分）: GPTs設計のポイントが3-4枚で伝わる
□ パート3（5分）: 実運用の結果が具体的に示せる
□ パート4（2分）: 振り返りと今後が明確

【内容チェック】

□ 数字が入っている（時間削減○%、使用回数○回など）
□ 対話ログのスクリーンショットがある
□ 改善前後の比較がある
□ 学びが言語化されている
```

---

# リハーサルのすすめ

### 本番前に1回は通す

```
【リハーサルでチェック】

1. 時間
   - 発表15分に収まるか
   - 各パートの配分は適切か

2. 内容
   - 専門用語を説明しているか
   - 論理の飛躍がないか

3. 資料
   - 文字が小さすぎないか
   - 図やスクリーンショットが見やすいか

【おすすめ】
・1人で声に出して練習（タイマー必須）
・可能なら同僚に聞いてもらう
```
