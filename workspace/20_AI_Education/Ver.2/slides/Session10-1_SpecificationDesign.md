# Part 1: なぜ仕様書が必要か

---

# GPTsは「作って終わり」ではない

### 運用フェーズで起きること

```
【仕様書なしの場合】

担当者A: 「このGPTs、誰が作ったんだっけ？」
担当者B: 「Instructionsって何が書いてあるの？」
新人C: 「エラーが出たけど、どう対応すればいい？」

→ 誰も答えられない
→ GPTsが放置される or 作り直し
```

> **問題**: 作成者しか分からない「属人化」が発生

---

# 仕様書の3つの目的

### ドキュメント化のメリット

| # | 目的 | 内容 |
|---|---|---|
| 1 | **引き継ぎ** | 作成者以外も理解・運用できる |
| 2 | **トラブル対応** | 問題発生時の対処方法が明確 |
| 3 | **継続的改善** | 変更履歴を追跡、品質を維持 |

---

# 仕様書がないとどうなるか

### よくある失敗パターン

| パターン | 問題 | 影響 |
|---|---|---|
| **属人化** | 作成者しか仕様を知らない | 担当者異動で運用停止 |
| **変更履歴なし** | 何をいつ変えたか不明 | 問題発生時の原因特定が困難 |
| **対応手順なし** | エラー時の対応が分からない | ユーザーからの問い合わせ対応が遅延 |
| **目的の忘却** | なぜ作ったか不明 | 不要な機能追加、本来の目的から逸脱 |

---

# 仕様書の読み手は誰か

### 3つのペルソナ

```
【ペルソナ1: 運用担当者】
- 日々のトラブル対応を担当
- 必要な情報: エラー対応、問い合わせ対応

【ペルソナ2: 後任の開発者】
- 機能追加・改善を担当
- 必要な情報: 設計意図、変更履歴、テスト方法

【ペルソナ3: 意思決定者】
- 継続利用の判断を担当
- 必要な情報: 目的、効果、コスト
```

---

# 仕様書作成のタイミング

### いつ作るか

```
Session 6: 企画書作成
         ↓
Session 7: Instructions作成
         ↓
Session 8: 知識ベース構築
         ↓

```

### 必須セクション

| # | セクション | 内容 |

# セクション1: 概要

### 目的・対象・機能を明確に

```markdown
## 1. 概要

### 1.1 GPTs名
提案書作成アシスタント

### 1.2 目的
営業担当者の提案書作成業務を効率化し、
作成時間を50%削減する

### 1.3 対象ユーザー
営業部門の社員（約50名）

### 1.4 主要機能
- 提案書のドラフト作成
- テンプレートに基づく整形
- 過去事例の参照・活用
```

### 必須項目と記載例

| 項目 | 必須 | 記載例 |
|---|---|---|
| GPTs名 | ✓ | 提案書作成アシスタント |
| 目的 | ✓ | 提案書作成時間を50%削減 |

| 主要機能 | ✓ | ドラフト作成、テンプレート整形 |
| 作成日 | ✓ | 2026年1月 |
| 作成者 | ✓ | 営業企画課 山田 |
| 想定利用頻度 | 推奨 | 週3-5回/人 |

# セクション2: Instructions

### 設定内容の全文を記載

```markdown
## 2. Instructions

### 2.1 全文
[Session 7で作成したInstructionsをそのまま記載]

### 2.2 構成要素の説明

| 要素 | 設計意図 |
|---|---|
| ペルソナ | 営業サポートの専門家として振る舞う |
| 対話フロー | Phase 1でヒアリング、Phase 2で生成 |
| 禁止事項 | 競合情報の提供を禁止 |
| 例外処理 | 範囲外は丁寧に断る |
```

---

# Instructionsセクションのポイント

### 設計意図を残す

```
【なぜ設計意図が必要か】

Instructions全文だけだと...
→ 「なぜこう書いたか」が分からない
→ 後任者が修正判断できない

設計意図を残すと...
→ 変更可否の判断が可能
→ 一貫性を保った改善ができる
```

> **例**: 「競合情報の禁止」は法務確認済み → 変更不可

# セクション3: 知識ベース構成

### ファイル一覧と更新ルール

```markdown
## 3. 知識ベース構成

### 3.1 ファイル一覧

| ファイル名 | 内容 | サイズ | 更新頻度 |
|---|---|---|---|
| template_proposal.md | 提案書テンプレート | 15KB | 年1回 |
| case_studies.json | 過去成功事例 | 120KB | 四半期 |
| product_pricing.csv | 価格表 | 8KB | 月次 |

### 3.2 更新ルール
- 価格表は毎月1日に更新
- 更新担当: 営業企画課
- 更新後はテストケースTC-01〜03を再実行
```

---

# 知識ベースセクションのポイント

### 更新担当と手順を明確に

| 項目 | 内容 |
|---|---|
| **ファイル一覧** | 名前・内容・サイズ・更新頻度 |
| **更新担当者** | 誰が更新するか |
| **更新手順** | どうやって更新するか |
| **更新後の確認** | 何をテストするか |
| **バックアップ** | 旧バージョンの保管 |

> **重要**: 更新手順が不明確だと、知識ベースが陳腐化する

# セクション4: 運用ガイドライン

### 日常運用のルール

```markdown
## 4. 運用ガイドライン

### 4.1 利用ルール
- 機密情報は入力しない
- 出力内容は必ず人間が確認
- 顧客提出前に上長承認を得る

### 4.2 利用時間
- 24時間利用可能
- サポート対応: 平日9:00-18:00

### 4.3 問い合わせ先
- 一次窓口: 営業サポートチーム
- 二次窓口: DX推進室
```

# 運用ガイドラインのポイント

### ユーザー向けと管理者向け

```
【ユーザー向け】
- 利用ルール（やっていいこと/ダメなこと）
- 利用時間
- 問い合わせ先
- よくある使い方

【管理者向け】
- 監視項目（利用状況、エラー率）
- 定期メンテナンス内容
- エスカレーションフロー
- 権限管理
```

---

# セクション5: トラブルシューティング

### よくある問題と対処法

```markdown
## 5. トラブルシューティング

### 5.1 よくある問題

| 問題 | 原因 | 対処法 |
|---|---|---|
| 回答が遅い | GPT-4使用時の負荷 | 時間をおいて再試行 |
| 情報が古い | Knowledge未更新 | 更新担当者に連絡 |
| 的外れな回答 | 入力が曖昧 | 具体的に入力し直す |
| エラー発生 | システム障害 | 30分後に再試行 |

### 5.2 エスカレーション
1. 営業サポートチーム（Slack #gpts-support）
2. DX推進室（緊急時のみ）
```

---

# トラブルシューティングのポイント

### ユーザーが自己解決できるように

```
【良いトラブルシューティング】
- 問題の症状が具体的
- 原因の説明がある
- 対処法がステップで書かれている
- 解決しない場合の連絡先がある

【悪いトラブルシューティング】
- 「エラーの場合は管理者に連絡」のみ
- 対処法が抽象的
- 連絡先が不明
```

# 運用開始判定のポイント

### なぜGo/No-Go判定が必要か

| 観点 | 説明 |
|---|---|
| **品質保証** | テストをすべてパスしているか確認 |
| **安全性** | 機密情報漏洩リスクがないか |
| **完成度** | ドキュメントが揃っているか |
| **責任明確化** | 誰が運用開始を承認したか記録 |

> **ポイント**: No-Goでも「要改善」として次アクションを明記

# セクション7: バージョン履歴

### 変更履歴を追跡可能に

```markdown
## 7. バージョン履歴

| Ver | 日付 | 変更内容 | 変更者 | 承認者 |
|---|---|---|---|---|
| 1.0 | 2026-01-10 | 初版リリース | 山田 | 佐藤課長 |
| 1.1 | 2026-02-01 | 価格表更新 | 山田 | - |
| 1.2 | 2026-03-15 | テンプレート改善 | 田中 | 佐藤課長 |
| 2.0 | 2026-04-01 | 機能追加（比較表生成） | 山田 | 佐藤課長 |

### 承認ルール
- 軽微な変更（Knowledge更新）: 承認不要
- 機能追加・Instructions変更: 課長承認
- 大規模変更: 部長承認
```

---

# バージョン履歴のポイント

### バージョン番号のルール

```
【セマンティックバージョニング（推奨）】

v X.Y.Z
  │ │ └── パッチ: バグ修正、軽微な変更
  │ └──── マイナー: 機能追加（後方互換あり）
  └────── メジャー: 大規模変更（後方互換なし）

例:
v1.0.0 → 初版リリース
v1.0.1 → Knowledge更新（パッチ）
v1.1.0 → テンプレート追加（マイナー）
v2.0.0 → Instructions全面改訂（メジャー）
```

# Part 3: 引き継ぎの観点

---

# 引き継ぎチェックリスト

### 他者が運用できるか

```
【引き継ぎチェックリスト】

□ GPTsの目的が明確に記載されているか
□ Instructionsの設計意図が説明されているか
□ Knowledge更新手順が明確か
□ トラブル対応手順があるか
□ 問い合わせ先が明記されているか
□ バージョン履歴が追跡可能か
□ 必要な権限が整理されているか
```

---

# 引き継ぎの失敗パターン

### よくある問題

| パターン | 問題 | 対策 |
|---|---|---|
| **暗黙知依存** | 「聞けば分かる」状態 | すべて文書化 |
| **情報散逸** | 複数ファイルに分散 | 1つの仕様書に統合 |
| **古い情報** | 更新されていない | 更新ルールを明確化 |
| **専門用語** | 用語が説明なし | 用語集を追加 |

---

# 良い引き継ぎ文書の特徴

### 5つの条件

| # | 条件 | 説明 |
|---|---|---|
| 1 | **自己完結** | 他の資料を参照しなくても理解できる |
| 2 | **構造化** | 目次・見出しで情報が整理されている |
| 3 | **具体的** | 抽象的でなく具体例がある |
| 4 | **最新** | 現状を正確に反映している |
| 5 | **検索可能** | キーワードで必要な情報が見つかる |

---

# 用語集の追加（推奨）

### 専門用語を説明

```markdown
## 付録: 用語集

| 用語 | 説明 |
|---|---|
| Instructions | GPTsに設定する動作指示 |
| Knowledge | GPTsに読み込ませる知識ファイル |
| Phase 1 | ヒアリングフェーズ |
| Phase 2 | 成果物生成フェーズ |
| 幻覚（Hallucination） | AIが事実と異なる情報を生成すること |
| エッジケース | 通常と異なる特殊な入力パターン |
```

# なぜメタプロンプトで仕様書を作るか

### メリット

| 観点 | 手動作成 | メタプロンプト生成 |
|---|---|---|
| **時間** | 2-3時間 | 30分 |
| **一貫性** | フォーマットがバラつく | 統一フォーマット |
| **網羅性** | セクション漏れのリスク | 必須項目を網羅 |
| **統合** | 手動でコピペ | 成果物を自動統合 |

---

```markdown
【共通DoD】
{品質基準}
```
