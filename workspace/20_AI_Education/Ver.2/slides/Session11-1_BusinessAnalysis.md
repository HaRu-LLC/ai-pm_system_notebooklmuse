# Part 1: 業務分析の基礎

---

# なぜ業務分析が必要か

### GPTs構築の前に「何を自動化するか」を決める

```
【よくある失敗パターン】

❌ 「AIで何かできそう」→ 作ってみる → 使われない
❌ 「とりあえずFAQ」→ 誰も質問しない → 放置
❌ 「便利そうなGPTs」→ 業務に合わない → お蔵入り

【成功パターン】

✅ 業務の課題を特定 → 適切なGPTsを設計 → 実際に使われる
```

---

# 業務分析の3ステップ

### Step 1 → Step 2 → Step 3

```
┌─────────────────────────────────────────────────┐
│             業務分析の3ステップ                  │
├─────────────────────────────────────────────────┤
│                                                 │
│  Step 1: 可視化                                 │
│     ↓   業務フローを図にする                    │
│                                                 │
│  Step 2: ボトルネック特定                       │
│     ↓   時間・品質・コストの課題を見つける      │
│                                                 │

```

# Step 1: 業務の可視化

### フロー図で「見える化」する

```
【業務フロー図の基本構造】

開始
  ↓
作業1（担当者A）──[成果物1]
  ↓
作業2（担当者B）──[成果物2]
  ↓
  ├── 条件分岐
  │     ├── Yes → 作業3a
  │     └── No  → 作業3b
  ↓
作業4（担当者C）──[最終成果物]
  ↓
終了
```

---

# フロー図の記載項目

### 各ステップで記録すべき情報

| 項目 | 記載内容 | 例 |
|---|---|---|
| **作業名** | 何をするか | 見積書を作成する |
| **担当者** | 誰がやるか | 営業担当 |
| **所要時間** | どれくらいかかるか | 30分/件 |
| **頻度** | どれくらい発生するか | 週20件 |
| **インプット** | 何が必要か | 顧客要望、価格表 |
| **アウトプット** | 何が出来るか | 見積書PDF |
| **ツール** | 何を使うか | Excel、社内システム |

---

# 可視化の例: 見積書作成業務

### テキスト形式のフロー図

```
【見積書作成業務フロー】

1. 顧客から見積依頼を受ける（営業）── 5分/件 ── 週20件
   ↓
2. 顧客要望をヒアリング（営業）── 15分/件
   ↓
3. 価格表から該当製品を検索（営業）── 10分/件
   ↓
4. Excelで見積書を作成（営業）── 30分/件 ← ボトルネック候補
   ↓
5. 上長承認（課長）── 待ち時間含め1日
   ↓
6. 顧客へ送付（営業）── 5分/件
```

---

# 可視化のポイント

### 粒度と範囲を意識する

| 観点 | 良い例 | 悪い例 |
|---|---|---|
| **粒度** | 1作業 = 1人が1回で完了 | 「書類作成」（曖昧すぎ） |
| **範囲** | 開始と終了が明確 | 終わりがない |
| **担当** | 具体的な役割 | 「誰か」 |
| **時間** | 実測値または目安 | 「たぶん30分」 |

> **コツ**: 最初は粗くても良い。AI（メタプロンプト）に詳細化してもらう

# Part 2: ボトルネックの特定

---

# ボトルネックとは

### 業務全体の流れを滞らせる箇所

```
【ボトルネックの3類型】

1. 時間のボトルネック
   → 1つの作業に時間がかかりすぎる
   → 例: 見積書作成30分/件

2. 品質のボトルネック
   → ミスや手戻りが多い
   → 例: 価格の入力ミスで再作成

3. 待ち時間のボトルネック
   → 承認や確認で待たされる
   → 例: 上長承認待ち1日
```

---

# ボトルネック発見の5つの質問

### 現場でよく使う質問パターン

| # | 質問 | 発見できる課題 |
|---|---|---|
| 1 | **一番時間がかかる作業は？** | 時間のボトルネック |
| 2 | **よく手戻りになる作業は？** | 品質のボトルネック |
| 3 | **待ち時間が長い作業は？** | 待ち時間のボトルネック |
| 4 | **属人化している作業は？** | 知識・スキルの偏り |
| 5 | **「面倒」と感じる作業は？** | 改善余地のある作業 |

---

# ボトルネック分析シート

### フロー図に課題を追記する

```markdown
| # | 作業 | 時間 | 頻度 | 課題 | 影響度 |
|---|---|---|---|---|---|
| 1 | 見積依頼受付 | 5分 | 週20件 | なし | - |
| 2 | ヒアリング | 15分 | 週20件 | 聞き漏れあり | 中 |
| 3 | 価格表検索 | 10分 | 週20件 | 最新版の確認に手間 | 中 |
| 4 | 見積書作成 | 30分 | 週20件 | ★手作業で時間大 | 高 |
| 5 | 上長承認 | 1日 | 週20件 | 待ち時間長い | 高 |
| 6 | 顧客送付 | 5分 | 週20件 | なし | - |
```

---

# 影響度の判断基準

### 高・中・低の3段階で評価

| 影響度 | 基準 | 例 |
|---|---|---|
| **高** | 週10時間以上の削減可能 または 重大なミスにつながる | 見積書作成（週10時間） |
| **中** | 週2-10時間の削減可能 または 軽微なミスにつながる | 価格表検索（週3時間） |
| **低** | 週2時間未満の削減可能 | 依頼受付（週1.5時間） |

> **計算例**: 30分/件 × 20件/週 = 10時間/週 → 影響度「高」

# AI適用の判断基準

### 「効果」×「実現性」で優先順位をつける

```
        高 ┌─────────────────┬─────────────────┐
           │                 │                 │
     効    │   △ 検討       │  ◎ 優先実施    │
     果    │  （効果高いが   │  （効果高く     │
           │   実現困難）    │   実現容易）    │
           │                 │                 │
           ├─────────────────┼─────────────────┤
           │                 │                 │
           │   × 見送り     │  ○ 余裕あれば  │
           │  （効果低く     │  （効果低いが   │
           │   実現困難）    │   実現容易）    │
           │                 │                 │
        低 └─────────────────┴─────────────────┘
                  低              高
                       実現性
```

---

# 効果の評価基準

### 時間削減・品質向上・コスト削減

| 観点 | 高効果 | 中効果 | 低効果 |
|---|---|---|---|
| **時間削減** | 50%以上削減 | 20-50%削減 | 20%未満 |
| **品質向上** | ミス90%削減 | ミス50%削減 | ミス30%未満 |
| **コスト削減** | 月10万円以上 | 月3-10万円 | 月3万円未満 |

> **例**: 見積書作成30分→10分（67%削減）= 高効果

---

# 実現性の評価基準

### データ・ツール・導入負荷

| 観点 | 高実現性 | 中実現性 | 低実現性 |
|---|---|---|---|
| **データ** | 既存データあり | 整備が必要 | 取得困難 |
| **ツール** | GPTsで完結 | API連携必要 | 専用開発必要 |
| **導入負荷** | すぐ使える | 研修が必要 | 組織変更必要 |

> **GPTsの強み**: Knowledge活用で「データあり→高実現性」になりやすい

---

# AI適用候補の評価例

### 見積書作成業務の場合

| 作業 | 課題 | 効果 | 実現性 | 優先度 |
|---|---|---|---|---|
| 見積書作成 | 手作業30分 | 高（67%減） | 高（テンプレ化可） | ◎ |
| 価格表検索 | 最新版確認 | 中（50%減） | 高（Knowledge化可） | ○ |
| ヒアリング | 聞き漏れ | 中（品質向上） | 中（チェックリスト化） | ○ |
| 上長承認 | 待ち時間 | 高（1日→即時） | 低（システム変更必要） | △ |

---

# GPTsで実現しやすい業務パターン

### AI適用の候補として優先すべきもの

| パターン | 例 | GPTsでの実現方法 |
|---|---|---|
| **定型文生成** | メール、報告書、議事録 | テンプレート + 変数 |
| **情報検索** | FAQ、マニュアル参照 | Knowledge活用 |
| **構造化** | 要約、分類、抽出 | プロンプト設計 |
| **チェック** | 校正、整合性確認 | ルールベース指示 |
| **ドラフト作成** | 企画書、提案書 | コンテキスト + 生成 |

---

# GPTsで実現しにくい業務パターン

### 別アプローチを検討すべきもの

| パターン | 理由 | 代替手段 |
|---|---|---|
| **リアルタイムデータ** | Knowledge更新が必要 | API連携（Actions） |
| **外部システム操作** | GPTs単体では不可 | RPA、連携ツール |
| **承認フロー** | 人の判断が必要 | ワークフローツール |
| **計算処理** | LLMは計算が苦手 | スプレッドシート連携 |
| **機密データ処理** | セキュリティリスク | オンプレLLM検討 |

# 業務分析シート（統合版）

### 1枚のシートで業務分析を完結

```markdown
# 業務分析シート

## 1. 業務概要
- 業務名:
- 担当者:
- 目的:
- 頻度:

## 2. 業務フロー
| # | 作業 | 担当 | 時間 | 頻度 | 課題 |
|---|---|---|---|---|---|

## 3. AI適用候補
| 作業 | 効果 | 実現性 | 優先度 | GPTs化の方針 |
|---|---|---|---|---|
```

---

# 業務分析の品質基準

### 良い分析と悪い分析の違い

| 観点 | 良い分析 | 悪い分析 |
|---|---|---|
| **具体性** | 「30分/件×週20件」 | 「時間がかかる」 |
| **網羅性** | 開始から終了まで | 一部だけ |
| **課題明確** | 原因と影響を記載 | 「なんとなく問題」 |
| **優先順位** | 効果×実現性で判断 | 全部やりたい |
| **アクション** | 具体的な次ステップ | 分析して終わり |

---

# よくある失敗パターンと対策

### 業務分析でつまずきやすいポイント

| 失敗パターン | 原因 | 対策 |
|---|---|---|
| フローが粗すぎ | 作業を大きく捉えすぎ | 1人1回で完了する単位に分解 |
| 時間が不明 | 計測していない | 1週間だけ計測してみる |
| 課題が曖昧 | 「なんとなく」で終わる | 5つの質問を使う |
| 全部やろうとする | 優先順位がない | 効果×実現性で絞る |
| 分析で終わる | 次ステップがない | GPTs設計に繋げる |

# Part 5: メタプロンプトでの業務分析

---

# なぜメタプロンプトで業務分析か

### AIを活用して分析の質と効率を上げる

| 従来のやり方 | メタプロンプト活用 |
|---|---|
| 自分でフロー図を書く | AIに生成してもらう |
| 経験則でボトルネック判断 | AIに分析してもらう |
| 知識がないと困る | AIが観点を提供 |
| 時間がかかる | 数分で初版完成 |
| 主観に偏りやすい | 客観的な視点を得る |

> **ポイント**: 最終判断は人間。AIは分析の「たたき台」を作る

---

# 業務分析メタプロンプトの構造

### 4つの構成要素

```
┌─────────────────────────────────────────────────┐
│        業務分析メタプロンプト                    │
├─────────────────────────────────────────────────┤
│ 1. 役割定義                                     │
│    → 業務改善コンサルタントとして振る舞う       │
├─────────────────────────────────────────────────┤
│ 2. 入力（業務概要）                             │
│    → 業務名・担当者・課題・作業内容            │
├─────────────────────────────────────────────────┤
│ 3. 出力形式                                     │
│    → フロー図・ボトルネック・期待改善効果      │
├─────────────────────────────────────────────────┤
│ 4. 分析観点                                     │
│    → 時間・品質・コスト・実現性                │
└─────────────────────────────────────────────────┘
```

---

# 業務分析メタプロンプト（基本形）

### EX-21で使用するテンプレート

```markdown
あなたは業務改善コンサルタントです。
以下の業務概要に基づき、業務分析を行ってください。

【業務概要】
業務名: {業務名}
担当者: {担当者}
現状の課題: {課題}
作業内容:
{作業内容を箇条書き}

【出力形式】
1. 業務フロー図（テキスト形式）
2. 各ステップの所要時間・頻度（推定含む）
3. ボトルネック分析（課題・影響度）
4. 期待される改善効果（時間/品質/コスト）
```

---

# 入力のコツ

### 良い入力と悪い入力

| 項目 | 良い入力例 | 悪い入力例 |
|---|---|---|
| 業務名 | 「月次営業レポート作成」 | 「レポート作成」 |
| 担当者 | 「営業部 各担当者→課長」 | 「営業」 |
| 課題 | 「毎月3日かかる、集計ミスが多い」 | 「大変」 |
| 作業内容 | 「売上データ収集、Excel集計、グラフ作成...」 | 「レポートを書く」 |

> **コツ**: 具体的に書くほど、AIの分析精度が上がる

---

# AIの出力を鵜呑みにしない

### 人間が確認すべきポイント

```
【AIの出力をレビューする観点】

□ フロー図は実態と合っているか？
  → 抜けている作業はないか
  → 順序は正しいか

□ 時間の推定は妥当か？
  → 実測値があれば補正する
  → 極端に短い/長い推定に注意

□ ボトルネックの指摘は正しいか？
  → 現場感覚と照らし合わせる
  → 見落としている課題はないか

□ AI適用候補の優先順位は妥当か？
  → 効果と実現性を自分で再評価
```

# EX-21: 業務分析メタプロンプト＋フロー図

自分の業務を分析し、メタプロンプトを使ってフロー図とボトルネック分析を生成する。

### 提出物

1. 業務分析メタプロンプト（自分の業務に合わせてカスタマイズ）
2. AI生成の業務フロー図（レビュー・修正済み）
3. ボトルネック分析（AIの出力＋自分の評価コメント）

# EX-21 タイムボックス

### 推奨時間配分

| 作業 | 推奨時間 | 上限 | 備考 |

| メタプロンプト作成 | 10分 | 15分 | テンプレートをカスタマイズ |

| **合計** | **55分** | **80分** | |

# EX-21の評価観点

### 合格ラインの目安

### どの業務を分析するか迷ったら

```
【選びやすい業務の特徴】

✅ 自分がよく知っている（毎日/毎週やる）
✅ 時間がかかっていると感じる
✅ 「もっと楽にならないか」と思っている
✅ 定型的な作業が含まれる
✅ テキストベースの作業が多い

【避けた方が良い業務】

❌ 自分があまり関わらない
❌ 機密性が非常に高い
❌ 外部システム操作が中心
❌ リアルタイムデータが必要
```
