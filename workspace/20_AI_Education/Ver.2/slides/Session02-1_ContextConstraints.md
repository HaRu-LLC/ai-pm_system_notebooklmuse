# Part 1: 文脈（Context）の設計

---

# 文脈がないとどうなる？

### 例: 「企画書を書いて」

AIの頭の中:
- 誰向け？（社内？顧客？）
- 何の企画？
- どんな会社？
- どのくらいの規模？

**→ 汎用的すぎて使えない回答になる**

---

# 文脈の3つの柱

```
┌─────────────────────────────────────────┐
│                                         │
│   1. 自分は誰か（ペルソナ）             │
│      └ 役職、経験、立場                 │
│                                         │
│   2. 相手は誰か（ターゲット）           │
│      └ 読み手、聞き手の属性             │
│                                         │
│   3. 状況は何か（シチュエーション）     │
│      └ 目的、背景、使用場面             │
│                                         │
└─────────────────────────────────────────┘
```

---

# 1. ペルソナ設定（自分は誰か）

**AIに「あなたは〇〇です」と役割を与える**

### 例
```
あなたは10年以上の経験を持つITコンサルタントです。
```

```
あなたは中小企業の経理担当者です。
```

```
あなたは新卒採用担当の人事マネージャーです。
```

---

# ペルソナの効果

### Before（ペルソナなし）
```
プロジェクト管理のコツを教えて
```
→ 一般的な教科書的回答

### After（ペルソナあり）
```
あなたは10年以上の経験を持つITプロジェクトマネージャーです。
新人PMに向けて、プロジェクト管理の実践的なコツを教えてください。
```
→ 実務経験に基づいた具体的なアドバイス

---

# ペルソナ設定パターン

| パターン | 例 | 効果 |
|---|---|---|
| 専門家 | 「あなたは〇〇の専門家です」 | 専門的な回答 |
| 経験者 | 「あなたは〇〇年の経験があります」 | 実践的な回答 |
| 立場 | 「あなたは〇〇部門の責任者です」 | 視点が明確に |
| 組み合わせ | 上記を組み合わせ | より精度が高く |

---

# 2. ターゲット設定（相手は誰か）

**出力を受け取る人を明確にする**

### 例
```
対象は新入社員（入社1年目）です。
```

```
読み手は経営層で、技術的な背景知識はありません。
```

```
クライアントは中小企業の社長で、ITに詳しくありません。
```

---

# ターゲットによる変化

### 同じ内容でも、ターゲットで表現が変わる

| ターゲット | 説明の特徴 |
|---|---|
| 専門家向け | 専門用語OK、詳細な技術説明 |
| 経営層向け | ビジネスインパクト重視、要点のみ |
| 初心者向け | 専門用語なし、具体例多め |
| 子供向け | たとえ話、簡単な言葉 |

---

# 3. シチュエーション（状況は何か）

**使用場面・目的・背景を伝える**

### 例
```
来週の経営会議で報告するための資料です。
```

```
新規顧客への提案プレゼンで使用します。
```

```
社内研修の講義資料として使用します。
```

---

# 文脈設計の実例

### Before
```
プレゼン資料を作って
```

### After
```
【自分】私は営業部のマネージャーです。
【相手】対象は新規顧客（製造業の中堅企業）です。
【状況】初回訪問での会社紹介プレゼンに使用します。
        持ち時間は15分です。
```

**→ 具体的で使える資料が生成される**

---

# Part 2: 制約（Constraints）の設計

---

# 制約がないとどうなる？

### 例: 「報告書を書いて」

AIの頭の中:
- どのくらいの長さ？
- 堅い文体？くだけた文体？
- 何を含める？何を除外する？

**→ 期待と違う長さ・トーンになりがち**

---

# 制約の5つのカテゴリ

```
┌─────────────────────────────────────────┐
│                                         │
│   1. 長さ（Length）                     │
│   2. トーン（Tone）                     │
│   3. 含める内容（Include）              │
│   4. 禁止事項（Exclude）                │
│   5. 形式（Format）                     │
│                                         │
└─────────────────────────────────────────┘
```

---

# 1. 長さの制約

| 指定方法 | 例 |
|---|---|
| 文字数 | 「300文字以内で」「500〜800文字で」 |
| 文数・段落数 | 「3文で」「2段落で」 |
| 時間 | 「1分で話せる長さで」「5分程度」 |
| ページ数 | 「A4 1枚に収まるように」 |
| 項目数 | 「3つのポイントで」「5項目以内で」 |

---

# 2. トーンの制約

| トーン | 使用場面 |
|---|---|
| フォーマル・敬語 | ビジネス文書、顧客向け |
| カジュアル | 社内チャット、ブログ |
| 熱意を込めて | 提案書、志望動機 |
| 客観的・論理的 | 報告書、分析レポート |
| 親しみやすく | FAQ、ヘルプ文書 |

---

# 3. 含める内容（Include）

**必ず入れてほしい要素を指定**

### 例
```
以下を必ず含めてください：
- 具体的な数値データ
- 実際の事例（2つ以上）
- 次のアクション
```

---

# 4. 禁止事項（Exclude）

**入れてほしくない要素を指定**

### 例
```
以下は含めないでください：
- 競合他社の名前
- 専門用語
- 否定的な表現
- 曖昧な表現（「〜かもしれない」等）
```

---

# 5. 形式の制約

| 形式 | 例 |
|---|---|
| 構造 | 「結論→理由→詳細の順で」 |
| 表現 | 「です・ます調で」「体言止めで」 |
| 言語 | 「日本語で」「英語と日本語併記で」 |
| 視点 | 「第三者視点で」「一人称で」 |

---

# 制約設計の実例

### Before
```
お礼メールを書いて
```

### After
```
【長さ】200文字程度
【トーン】丁寧な敬語、感謝の気持ちが伝わるように
【含める】・商談への謝意 ・次回アクションの確認
【禁止】・価格交渉への言及 ・曖昧な期日
【形式】件名と本文を分けて出力
```

---

# 文脈と制約の組み合わせ

```
【ペルソナ】あなたは経験10年のベテラン営業です。
【ターゲット】読み手は先日商談した取引先の部長です。
【状況】商談翌日のフォローアップメールです。

【長さ】200文字程度
【トーン】丁寧な敬語、熱意が伝わるように
【禁止】具体的な価格・見積金額への言及
【形式】件名と本文を分けて
```

**→ 状況に完璧にマッチしたメールが生成される**

---

# よくある失敗パターン

| 失敗 | 改善 |
|---|---|
| 制約が多すぎる | 重要な3〜5個に絞る |
| 矛盾する制約 | 優先順位を明示 |
| 曖昧な制約 | 具体的な数値・例示を使う |
| 制約なし | 最低限「長さ」「トーン」は指定 |

---

# 2-1 まとめ

### 文脈の3つの柱
- **ペルソナ**: 自分は誰か
- **ターゲット**: 相手は誰か
- **シチュエーション**: 状況は何か

### 制約の5カテゴリ
- 長さ・トーン・含める・禁止・形式

**文脈と制約で、AIの出力精度は劇的に上がる**

# 今週の実習課題 EX-03

---

# EX-03: ペルソナ・制約設計【必須】

**課題内容**:
同じ「議事録作成」タスクに対して、異なるペルソナ・制約を設定し、出力の違いを比較する

**成果物**:
- 3種類以上のペルソナ・制約パターン
- 各パターンのプロンプトと応答結果
- 違いの分析コメント

**評価基準**:
3種類以上のバリエーションがあり、違いを説明できている

---

# EX-03 の取り組み方

### ステップ1: ベースタスクを決める
「会議メモから議事録を作成する」

### ステップ2: 3パターン以上を設計

| # | ペルソナ | ターゲット | 制約 |
|---|---|---|---|
| A | 新入社員 | 上司 | フォーマル、詳細に |
| B | ベテラン秘書 | 経営層 | 要点のみ、1ページ |
| C | プロジェクトリーダー | チームメンバー | アクション重視 |

### ステップ3: 実行して違いを分析

---

# 課題のポイント

### やってみよう
1. まず1パターン作って実行
2. 「ここを変えたらどうなる？」と仮説を立てる
3. 変えて実行、違いを観察

### 困ったら
- **テンプレートを使う**（ポータルからダウンロード可能）
- 極端な設定（超カジュアル vs 超フォーマル）で違いを際立たせる
- 比較しやすいよう同じ入力データを使う
