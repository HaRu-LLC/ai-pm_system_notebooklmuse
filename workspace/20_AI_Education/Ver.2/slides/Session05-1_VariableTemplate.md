# Session 4で作った複合プロンプト

```
あなたは企画書作成のベテランコンサルタントです。

【Phase 1: 要件定義】
以下のサイクルを繰り返してください：
1. まとめ: 私が伝えた企画の要点を整理
2. 提案: 企画書の構成案を提示
3. 質問: 目的・読者・必須要素・制約・重視点について確認

私が「OK」と言ったらPhase 2に進んでください。

【Phase 2: 企画書作成】
確定した要件に基づいて企画書を作成してください。
作成後、改善サイクルを繰り返してください。
私が「完了」と言うまで続けてください。
```

---

# 複合プロンプトの課題

### 毎回作り直す必要がある

| 用途 | 必要な作業 |
|---|---|
| 企画書作成 | 複合プロンプトを設計 |
| 提案書作成 | また複合プロンプトを設計 |
| 議事録作成 | また複合プロンプトを設計 |
| メール作成 | また複合プロンプトを設計 |

**→ 構造は同じなのに、毎回書き直している...**

---

# 解決策: 変数化

### 変わる部分と変わらない部分を分離

| 種類 | 内容 | 例 |
|---|---|---|
| **固定部分** | 構造・動作・フロー | Phase 1→Phase 2、改善ループ |
| **固定部分** | 品質管理 | **DoD（品質基準）、ガードレール** |
| **変数部分** | 用途に応じて変わる | ペルソナ、目的、成果物、制約 |

**→ 変数を差し替えるだけで、様々な用途に対応！**

# Part 2: 変数化とは

---

# 変数化の基本

### プログラミングの「変数」と同じ考え方

```
【変数化前】
あなたは企画書作成のベテランコンサルタントです。

【変数化後】
あなたは{役割}です。
```

| 変数名 | 用途例1 | 用途例2 | 用途例3 |
|---|---|---|---|
| {役割} | 企画書作成のコンサル | 営業提案のプロ | 議事録作成の達人 |

---

# 主要な変数

### 複合プロンプトで変数化すべき項目

| 変数名 | 内容 | 例 |
|---|---|---|
| **{役割}** | ペルソナ | 企画書のプロ、営業のベテラン |
| **{目的}** | 何を達成するか | 企画書作成、提案書作成 |
| **{成果物}** | 最終的なアウトプット | A4 1枚の企画書、5分プレゼン資料 |
| **{制約}** | 条件・ルール | 専門用語を避ける、500文字以内 |
| **{確認観点}** | Phase 1で聞くこと | 目的、読者、必須要素... |

> ⚠️ **注意**: DoD（品質基準）とガードレールは**固定項目**として扱い、変数化しない

---

# 変数化の例

### Before: 特定用途のプロンプト

```
あなたは企画書作成のベテランコンサルタントです。
私が企画書を作成できるよう支援してください。
企画書はA4で3ページ程度、経営層向けで作成してください。
```

### After: 変数化されたテンプレート

```
あなたは{役割}です。
私が{目的}を達成できるよう支援してください。
{成果物}は{制約}で作成してください。
```

# 汎用テンプレートの構造

```
あなたは{役割}です。

【Phase 1: 要件定義】
以下のサイクルを繰り返してください：
1. まとめ: 私が伝えた内容を整理
2. 提案: {成果物}の構成案を提示
3. 質問: {確認観点}＋品質基準（DoD）について確認

私が「OK」と言ったらPhase 2に進んでください。

【Phase 2: {目的}】
確定した要件に基づいて{成果物}を作成してください。
{制約}を守ってください。
作成後、改善サイクルを繰り返してください。
私が「完了」と言うまで続けてください。

⚠️ 機密情報は入力しないでください
⚠️ AIの出力は事実確認が必要です
```

# テンプレート設計のコツ

### 1. 変数は多すぎない

```
✅ 良い例: 5-7個の変数
   {役割}, {目的}, {成果物}, {制約}, {確認観点}

❌ 悪い例: 15個以上の変数
   → 埋めるのが大変、ミスが増える
```

---

# テンプレート設計のコツ

### 2. 変数名は直感的に

```
✅ 良い例
   {役割} → 何を入れるか明確
   {成果物} → アウトプットだとわかる

❌ 悪い例
   {var1}, {param_a} → 何を入れるか不明
```

---

# テンプレート設計のコツ

### 3. デフォルト値を用意する

```
【変数定義】
{役割} = ビジネスコンサルタント（デフォルト）
{確認観点} = 目的、読者、必須要素、制約、優先度（デフォルト）

→ 埋め忘れても動作する
→ 必要に応じてカスタマイズ
```

---

# テンプレート設計のコツ

### 4. 動作確認用の例を用意する

```
【テスト用変数セット】
{役割} = テスト用ペルソナ
{目的} = テスト用タスク
{成果物} = テスト用アウトプット
{制約} = テスト用制約
{確認観点} = テスト用観点

→ テンプレートが正しく動くか確認できる
```

# Part 5: まとめと課題

---

# 固定要素と変数要素の整理

### テンプレート設計の基本原則

| 分類 | 項目 | 理由 |
|---|---|---|
| **固定要素** | Phase構造（Phase 1→Phase 2） | 動作の骨格 |
| **固定要素** | 改善ループ（まとめ→提案→質問） | 品質向上の仕組み |
| **固定要素** | DoD（品質基準）の確認 | 完成判断の基準 |
| **固定要素** | ガードレール（注意事項） | 業務利用の安全対策 |
| **変数要素** | {役割}, {目的}, {成果物}, {制約}, {確認観点} | 用途で変わる |

---

# 共通DoD（品質基準）プリセット

### どの用途でも使える最低限の完成基準

```
【共通DoD（品質基準）】
1. 目的達成度: 依頼した目的を満たしているか
2. 完成度: 必要な要素がすべて揃っているか
3. 実用性: そのまま業務で使える状態か
4. 正確性: 事実・数値に誤りがないか
```

> ⚠️ DoDは**変数化せず固定**。用途固有の基準があれば追加する

---

# 5-1 まとめ

### 変数化とは
- プロンプトの「変わる部分」を `{変数名}` で表現
- 固定部分（構造・フロー・DoD・ガードレール）と変数部分を分離

### テンプレート設計のポイント
- 主要変数: {役割}, {目的}, {成果物}, {制約}, {確認観点}
- **DoD・ガードレールは固定項目**（変数化しない）
- 変数は5-7個程度に抑える
- 変数名は直感的に
- デフォルト値と動作確認例を用意

# EX-09: 汎用プロンプトテンプレート（Phase 2成果物）

### 課題内容

Session 4で作成した複合プロンプトを変数化し、
汎用テンプレートを作成してください。
そのテンプレートを3つの異なる用途で使用してください。

### 提出物
1. 汎用プロンプトテンプレート（変数化されたもの）
2. 変数定義シート（各変数の説明）
3. 3用途での実行例（変数設定＋対話ログ）
4. 振り返り（テンプレート化のメリット・課題）

---

# EX-09 取り組みのポイント

### 成功のコツ

1. **Session 4のプロンプトをベースにする**
   - 動作確認済みの複合プロンプトを変数化

2. **3用途は異なる分野を選ぶ**
   - 例: 企画書、メール、議事録
   - 同じ分野の3パターンは避ける

3. **変数定義シートを丁寧に作る**
   - 各変数の説明と入力例を記載
