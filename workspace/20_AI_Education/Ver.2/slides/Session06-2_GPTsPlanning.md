# Part 1: GPTs企画メタプロンプトの設計

---

# GPTs企画メタプロンプトとは

### 「GPTs企画書を生成するプロンプト」

```
【従来のアプローチ】
自分で企画書を書く
↓
「このGPTsは〇〇のために...」と手書き
↓
抜け漏れ、時間がかかる

【メタプロンプトアプローチ】
業務情報を入力するだけ
↓
AIが企画書を生成
↓
必要に応じて修正・改善
```

---

# GPTs企画書の必須項目

### 企画書に含めるべき6要素

| 要素 | 内容 | 例 |
|---|---|---|
| **GPTs名** | わかりやすい名前 | 「営業提案ドラフター」 |
| **目的** | 何を解決するか | 提案書作成時間を50%削減 |
| **機能概要** | 主な機能 | 顧客情報→提案書骨子 |
| **対話フロー** | 想定される会話の流れ | ヒアリング→提案→修正 |
| **知識ベース** | 必要なファイル | 商品カタログ、テンプレート |
| **成功指標** | 効果測定の基準 | 作成時間、採用率 |

---

# メタプロンプト設計の5ステップ

### Phase 3の共通パターン

| Step | 内容 | 成果物 |
|---|---|---|
| 1 | 目的を明確にする | 「GPTs企画書を生成する」 |
| 2 | 入力変数を設計する | {対象業務}, {課題}, {ユーザー} |
| 3 | 出力形式を設計する | 企画書の構成項目 |
| 4 | ガードレールを追加する | 品質基準、禁止事項 |
| 5 | 改善ループを組み込む | 確認→修正のサイクル |

---

# Step 1: 目的を明確にする

### GPTs企画メタプロンプトの目的

```
【このメタプロンプトの目的】
業務の概要情報を入力として受け取り、
GPTs企画書を生成する。

【期待する効果】
- 企画書作成の時間短縮
- 必要項目の抜け漏れ防止
- 品質の標準化
```

---

# Step 2: 入力変数を設計する

### GPTs企画に必要な入力

| 変数名 | 内容 | 入力例 |
|---|---|---|
| **{対象業務}** | 対象とする業務 | 営業提案書の作成 |
| **{課題}** | 解決したい問題 | 作成に3時間かかる |
| **{ユーザー}** | 想定利用者 | 営業チーム10名 |
| **{既存リソース}** | 活用可能な資料 | 商品カタログ、過去提案書 |
| **{成功基準}** | 達成したい目標 | 作成時間50%削減 |

---

# Step 3: 出力形式を設計する

### GPTs企画書の構成

```
【GPTs企画書の出力形式】

## 1. GPTs名
[わかりやすい名前]

## 2. 目的・解決する課題
[何をどう改善するか]

## 3. 機能概要
[主な機能リスト]

## 4. 想定される対話フロー
[ユーザーとGPTsのやり取りイメージ]

## 5. 必要な知識ベース
[アップロードするファイル一覧]

## 6. 成功指標
[効果測定の基準]
```

---

# 成功指標の設計ポイント

### 測定可能な指標を設定する

| 指標 | 現状値 | 目標値 | 測定方法 |
|---|---|---|---|
| 作成時間 | 180分/件 | 90分/件 | タイムスタンプ記録 |
| 提案採用率 | 15% | 25% | CRM成約データ |
| 手戻り率 | 30% | 10% | 修正依頼回数 |

> **ポイント**: 「効率化」「品質向上」ではなく、**数値**で定義する

---

# Step 4: ガードレールを追加する

### GPTs企画生成時の注意点

```
【ガードレール（プロンプト本文に記載）】
⚠️ 機密情報は入力しないでください
⚠️ AIの出力は事実確認が必要です

【品質基準（DoD）】
1. 目的達成度: 入力された課題を解決できる設計か
2. 完成度: 6項目すべてが具体的に記述されているか
3. 実用性: このまま実装に進める詳細度か
4. 正確性: 技術的に実現可能な内容か

【業務移管チェックリスト（別途確認）】
□ 機密区分: 社外秘情報を入力していないか
□ 事実確認: 記載内容の実現可能性を確認したか
□ 責任者: GPTs運用の責任者は明確か
□ 再利用可否: 他者が使ってよいか確認したか
```

---

# Step 5: 改善ループを組み込む

### Phase 1 + Phase 2構造

```
【Phase 1: 要件ヒアリング】

【Phase 2: 企画書生成】

以下のサイクルを繰り返してください：
1. まとめ: 現在の企画書の概要を説明
2. 提案: 改善できる点を提示
3. 質問: 修正の方向性を確認

ユーザーが「完了」と言うまで続ける
```

# Part 4: よくある失敗と対策

---

# 失敗パターン1: 業務情報が曖昧

### 問題
- 「提案書を作りたい」だけでは情報不足
- AIが質問しても「適当に考えて」と返す

### 対策
```
【入力前チェックリスト】
□ 対象業務は具体的か？（「提案書」→「BtoB製造業向け提案書」）
□ 課題は数値化されているか？（「遅い」→「3時間かかる」）
□ ユーザーは特定されているか？（「みんな」→「営業10名」）
□ 成功基準は測定可能か？（「効率化」→「50%削減」）
```

---

# 失敗パターン2: 企画が壮大すぎる

### 問題
- 「すべてを自動化したい」
- 「AIが完璧に判断してほしい」

### 対策
```
【スコープを絞る質問】
- 最も時間がかかっている作業は何か？ → それだけに絞る
- 今日から使える最小機能は何か？ → MVP（最小限の実装）を定義
- 人間の判断が必要な部分はどこか？ → AIにさせない

【良い例】
× 「営業プロセス全体を自動化するGPTs」
○ 「初回提案書の骨子を30分で作るGPTs」
```

---

# 失敗パターン3: 知識ベースが未整理

### 問題
- 「既存資料はあるけどバラバラ」
- 「PDFが100個ある」

### 対策
```
【知識ベースの事前整理】
1. 本当に必要なファイルを選定（20ファイル以内）
2. カテゴリ別に整理（商品情報、FAQ、テンプレート等）
3. 最新版に統一（古い資料は除外）
4. 検索しやすい見出し構造にする

【Phase 3での対応】
Session 8「知識ベースの構築」で詳しく学ぶ
→ 今は「必要そうなファイル名」をリストすればOK
```

# EX-12: GPTs企画書（AI生成）

### 課題内容

EX-11で作成したメタプロンプトを使って、

# EX-12 取り組みのポイント

### 成功のコツ

1. **自分の業務を題材にする**
   - 実際に作りたいGPTsを考える
   - Phase 4の最終課題につなげる

2. **業務情報を具体的に入力する**
   - 数値、固有名詞を使う
   - 曖昧な表現を避ける

3. **改善ループを回す**
   - 最初の生成で満足しない
   - 2-3回の修正イテレーションを行う

4. **企画書の実現可能性を確認する**
   - 知識ベースは用意できるか
   - 成功指標は測定できるか
