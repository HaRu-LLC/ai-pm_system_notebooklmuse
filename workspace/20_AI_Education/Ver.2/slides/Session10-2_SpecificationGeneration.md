
# メタプロンプト: 品質基準（DoD）

### 必須の品質要件

```markdown
【共通DoD（品質基準）】

生成する仕様書は以下を満たすこと：

1. 自己完結性
   - 他の資料を参照せず理解できる
   - 専門用語は説明付き

2. 構造化
   - 目次・見出しで情報が整理されている
   - 必要な情報にすぐアクセスできる

3. 具体性
   - 抽象的でなく具体例がある
   - 手順はステップで記載

4. 引継可能性
   - 作成者以外が運用できる詳細度
   - 連絡先・担当者が明記されている

5. 実用性
   - このまま運用に使える品質
   - コピペで設定・実行できる
```

# 実行の流れ

### ステップバイステップ

```
Step 1: 成果物を準備
        ↓
Step 2: メタプロンプトに成果物を入力
        ↓
Step 3: AIに実行させる
        ↓
Step 4: 生成結果を確認
        ↓
Step 5: 不足・誤りを修正依頼
        ↓
Step 6: 完成版をダウンロード
```

---

# Step 2: 成果物の入力

### 効率的な入力方法

```
【入力のコツ】

1. 成果物をそのままコピペ
   - フォーマットを崩さない
   - 省略しない（特にInstructions）

2. 見出しで区切る
   - 「## 1. GPTs企画書」など
   - AIが構造を理解しやすくなる

3. 長すぎる場合
   - 分割して入力
   - 「続きを入力します」と伝える
```

---

# Step 3: AIに実行させる

### 実行時のポイント

```
【実行時のコツ】

1. 一度に全部生成
   - 部分生成だと一貫性が崩れる
   - 長くなっても全体を生成させる

2. 生成中の確認
   - 途中で止まったら「続けて」
   - 省略されたら「省略せず全文」

3. 対話で改善
   - 「セクション3をもう少し詳しく」
   - 「トラブルシューティングを5件追加」
```

# よくある問題と修正依頼

### 対話で改善する

| 問題 | 修正依頼の例 |
|---|---|
| Instructions省略 | 「Instructionsを省略せず全文記載してください」 |
| セクション不足 | 「セクション5トラブルシューティングが抜けています」 |
| 抽象的 | 「運用ガイドラインをより具体的に書いてください」 |
| 情報不足 | 「知識ベースの更新手順を追加してください」 |
| 形式不統一 | 「すべての表を同じフォーマットにしてください」 |

---

# 修正依頼の例

### 具体的な指示

```markdown
【修正依頼1】
セクション2のInstructionsが一部省略されています。
EX-14の全文をそのまま記載してください。

【修正依頼2】
トラブルシューティングが3件しかありません。
以下の問題も追加してください：
- 回答が長すぎる場合
- 日本語以外の入力があった場合

【修正依頼3】
知識ベースの更新手順が抽象的です。
以下の形式でステップを具体化してください：
1. 〇〇を開く
2. △△をクリック
3. □□をアップロード
```
