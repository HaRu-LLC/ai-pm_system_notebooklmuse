## 1. 最終発表概要

### 1.1 発表形式

| 項目 | 内容 |
|---|---|
| **発表時間** | 15分（発表）+ 5分（質疑応答）= 20分 |
| **発表形式** | スライドプレゼンテーション＋デモ（任意） |
| **スライド枚数** | 10-15枚程度 |
| **発表対象** | 講師＋同期受講者 |
| **評価者** | 講師（メイン）＋同期受講者（相互評価） |

### 1.2 発表構成（4パート）

| パート | 内容 | 目安時間 | スライド枚数 |
|---|---|---|---|
| **1. 対象業務と課題** | 業務概要、課題・ペインポイント | 3分 | 2-3枚 |
| **2. GPTs設計・実装** | GPTs概要、設計プロセス、Instructions概要 | 5分 | 4-5枚 |
| **3. 実運用の結果** | 運用方法、対話ログ、効果と課題、改善 | 5分 | 4枚 |
| **4. 振り返りと今後** | 学んだこと、今後の計画、まとめ | 2分 | 2-3枚 |

### 1.3 評価構成（100点満点）

| 評価種別 | 配点 | 評価者 | 評価タイミング |
|---|---|---|---|
| **講師評価** | 80点 | 講師 | 発表中・発表後 |
| **相互評価** | 20点 | 同期受講者 | 発表直後 |
| **Gemini API評価** | 参考値 | システム | スライド提出時 |

## 2. 講師評価シート（80点満点）

### 2.1 評価観点と配点

| # | 評価観点 | 配点 | 説明 |
|---|---|:---:|---|
| 1 | **対象業務の適切さ** | 12点 | GPTs化に適した業務を選定しているか |
| 2 | **設計・実装の質** | 16点 | メタプロンプト活用、仕様書完成度 |
| 3 | **運用実績** | 16点 | 実際に使用した証跡、フィードバック収集 |
| 4 | **改善サイクル** | 16点 | フィードバックに基づく改善実施 |
| 5 | **発表の質** | 12点 | 構成、わかりやすさ、時間管理 |
| 6 | **学びの深さ** | 8点 | Phase 1-4全体の振り返り |

### 2.2 評価ルーブリック詳細

#### 2.2.1 対象業務の適切さ（12点）

| レベル | 点数 | 基準 |
|---|:---:|---|
| **優秀** | 11-12点 | GPTs化に最適な業務を選定。Session 11の分析に基づき、効果と実現性のバランスが取れている |
| **良好** | 9-10点 | 適切な業務選定。GPTs化の効果が見込める |
| **合格** | 7-8点 | 基本的な業務選定ができている |
| **要改善** | 4-6点 | 選定理由が不明確、またはGPTs化に不向きな業務 |
| **不足** | 0-3点 | 業務選定の根拠がない、または未選定 |

#### 2.2.2 設計・実装の質（16点）

| レベル | 点数 | 基準 |
|---|:---:|---|
| **優秀** | 14-16点 | Instructions設計が詳細で実用的。知識ベースが適切に構成されている。メタプロンプトを効果的に活用 |
| **良好** | 11-13点 | 設計が具体的で、実際に動作するGPTsが構築されている |
| **合格** | 8-10点 | 基本的な設計・実装ができている |
| **要改善** | 4-7点 | 設計が不完全、または動作に問題がある |
| **不足** | 0-3点 | 設計・実装ができていない |

#### 2.2.3 運用実績（16点）

| レベル | 点数 | 基準 |
|---|:---:|---|
| **優秀** | 14-16点 | 10回以上の実業務での使用記録。詳細なログ（入力/出力/評価/気づき）。多様な利用シーン |
| **良好** | 11-13点 | 7回以上の使用記録。ログが具体的。実業務で使用 |
| **合格** | 8-10点 | 5回以上の使用記録。基本的なログがある |
| **要改善** | 4-7点 | 使用回数不足、またはログが不十分 |
| **不足** | 0-3点 | 運用実績がない、または記録がない |

#### 2.2.4 改善サイクル（16点）

| レベル | 点数 | 基準 |
|---|:---:|---|
| **優秀** | 14-16点 | 複数回の改善を実施。Before/Afterが明確。改善効果を検証済み。継続的改善の姿勢 |
| **良好** | 11-13点 | 2回以上の改善。Before/Afterを記録。効果を確認 |
| **合格** | 8-10点 | 1回以上の改善を実施。基本的な記録がある |
| **要改善** | 4-7点 | 改善が不十分、または記録が曖昧 |
| **不足** | 0-3点 | 改善を実施していない |

#### 2.2.5 発表の質（12点）

| レベル | 点数 | 基準 |
|---|:---:|---|
| **優秀** | 11-12点 | 4パート構成が明確。15分以内で完結。デモやスクリーンショットを効果的に使用。聴衆を意識した説明 |
| **良好** | 9-10点 | 構成が適切。時間管理ができている。わかりやすい説明 |
| **合格** | 7-8点 | 基本的な構成と時間管理ができている |
| **要改善** | 4-6点 | 構成が不明確、または時間超過 |
| **不足** | 0-3点 | 発表ができていない |

#### 2.2.6 学びの深さ（8点）

| レベル | 点数 | 基準 |
|---|:---:|---|
| **優秀** | 7-8点 | Phase 1-4全体を通じた成長を具体的に言語化。技術面・プロセス面の両方の学び。今後の活用方針が明確 |
| **良好** | 5-6点 | 学びを言語化。今後の計画がある |
| **合格** | 4点 | 基本的な振り返りができている |
| **要改善** | 2-3点 | 振り返りが浅い |
| **不足** | 0-1点 | 振り返りがない |

## 3. 相互評価シート（20点満点）

### 3.1 評価観点と配点

| # | 評価観点 | 配点 | 説明 |
|---|---|:---:|---|
| 1 | **業務課題の明確さ** | 5点 | 課題が具体的でわかりやすかったか |
| 2 | **GPTsの有用性** | 5点 | 実際に使えそう・使いたいと思ったか |
| 3 | **発表のわかりやすさ** | 5点 | デモ・説明がわかりやすかったか |
| 4 | **学びになった点** | 5点 | 自分の参考になる点があったか |

### 3.2 評価基準

各項目5点満点で以下の基準で評価:

| 点数 | 基準 |
|:---:|---|
| 5点 | 非常に良い。大いに参考になった |
| 4点 | 良い。参考になった |
| 3点 | 普通。まあまあ良かった |
| 2点 | やや不十分。もう少し工夫が欲しい |
| 1点 | 不十分。改善が必要 |

### 3.4 相互評価の集計方法

1. 全評価者の点数を集計
2. 最高点と最低点を除外（参加者4名以上の場合）
3. 残りの平均を算出
4. 20点満点に換算

**計算例**:
- 評価者A: 18点、評価者B: 16点、評価者C: 19点、評価者D: 14点
- 最高（19点）と最低（14点）を除外
- (18 + 16) ÷ 2 = 17点

## 4. Gemini API評価（参考値）

### 4.1 評価対象

Webアプリに提出された以下の成果物を自動評価:

| 課題 | 評価対象 |
|---|---|
| **EX-26** | 発表スライド（PDF/PPTX）+ Phase 1-4全体振り返り（テキスト） |

### 4.2 評価プロンプト

```
あなたは教育評価の専門家です。以下の「最終発表資料」課題を評価してください。

## 評価対象
- 最終発表スライド（10-15枚程度）
- Phase 1-4全体振り返り

## 評価基準（各20点、合計100点）

1. **スライド構成（20点）**
   - 4パート構成（対象業務・設計・運用・振り返り）に沿っているか
   - 10-15枚程度で適切なボリュームか
   - 15分発表の時間配分が計画されているか

2. **内容の具体性（20点）**
   - 数字・データが入っているか（時間削減○%等）
   - スクリーンショットや対話ログがあるか
   - 具体的な事例が示されているか

3. **学びの言語化（20点）**
   - 技術面・プロセス面の学びが明確か
   - 改善サイクルを通じた成長が伝わるか
   - 今後の活用計画があるか

4. **発表資料の質（20点）**
   - スライドが見やすいか
   - 情報量が適切か
   - デザインが統一されているか

5. **全体振り返り（20点）**
   - Phase 1-4を通じた成長が伝わるか
   - 「作って終わり」ではなく「使って改善」を体験したか
   - 今後の業務活用方針があるか
```

### 4.3 Gemini API評価の位置づけ

| 項目 | 説明 |
|---|---|
| **用途** | 参考値として講師に提供。最終評価には含めない |
| **理由** | 発表の質（話し方・時間管理・聴衆とのインタラクション）はLLMで評価困難 |
| **表示** | 講師ダッシュボードに「AI参考評価」として表示 |

## 5. 総合評価算出

### 5.1 スコア算出式

```
総合評価 = 講師評価（80点満点） + 相互評価（20点満点）
```

### 5.2 評価レベル

| 総合点 | レベル | 備考 |
|---|---|---|
| 90-100点 | **優秀** | 修了証「優秀」として記載 |
| 80-89点 | **良好** | 修了証発行 |
| 70-79点 | **合格** | 修了証発行 |
| 60-69点 | **条件付き合格** | 改善レポート提出で合格 |
| 0-59点 | **要再発表** | 再発表の機会を提供 |

### 5.3 修了要件（最終発表関連）

| # | 要件 | 必須 |
|---|---|:---:|
| 1 | 最終発表を実施 | ✓ |
| 2 | 総合評価60点以上 | ✓ |
| 3 | 質疑応答に回答 | ✓ |

## 6. 評価フロー

### 6.1 発表当日の流れ

```
1. 発表準備（5分）
   └─ 発表者がスライドを投影
   └─ 講師・参加者が評価シートを準備

2. 発表（15分）
   └─ 発表者がプレゼンテーション
   └─ 講師・参加者がメモを取る

3. 質疑応答（5分）
   └─ 講師・参加者からの質問
   └─ 発表者が回答

4. 評価記入（3分）
   └─ 講師が講師評価シートを記入
   └─ 参加者が相互評価シートを記入（Webアプリ）

5. 次の発表へ（2分休憩）
```

### 6.2 評価入力タイミング

| 評価種別 | 入力タイミング | 入力方法 |
|---|---|---|
| **講師評価** | 全発表終了後 | Webアプリ管理画面 |
| **相互評価** | 各発表直後（3分） | Webアプリ（受講者画面） |
| **Gemini API評価** | スライド提出時（自動） | システム自動実行 |

### 6.3 結果集計・通知

```
1. 発表会終了後
   └─ 講師評価を入力（当日中）
   └─ 相互評価を集計（自動）

2. 翌営業日
   └─ 総合評価を算出
   └─ 受講者にメール通知
   └─ ダッシュボードに反映

3. 条件付き合格の場合
   └─ 改善レポート提出を依頼
   └─ 提出後に再評価
```

## 8. フィードバック方法

### 8.1 個別フィードバック

| 項目 | 内容 | タイミング |
|---|---|---|
| **総合スコア** | 100点満点での評価 | 翌営業日メール |
| **観点別スコア** | 6観点の内訳 | ダッシュボード |
| **講師コメント** | 良かった点、改善へのアドバイス | ダッシュボード |
| **相互評価サマリー** | 参加者からのフィードバック（匿名） | ダッシュボード |

### 8.2 全体フィードバック（発表会終了時）

発表会の最後に講師が以下を共有:

1. **全体傾向**
   - 良かった点（共通して見られた強み）
   - 改善機会（共通して見られた課題）

2. **ベストプラクティス共有**
   - 特に参考になる事例の紹介
   - 効果的だった工夫の共有

3. **今後のアドバイス**
   - 業務でのAI活用継続に向けて
   - 学習コミュニティの紹介

## 10. チェックリスト

### 10.1 発表者向けチェックリスト

```
【発表前】
□ スライドを10-15枚に収めた
□ 4パート構成になっている
```
