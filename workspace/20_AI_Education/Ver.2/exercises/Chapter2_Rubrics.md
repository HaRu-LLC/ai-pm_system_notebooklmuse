### 2.2 LLM評価プロンプト仕様（Gemini API用）

> [!NOTE]
> 専用WebアプリからGemini APIを呼び出し、Structured Outputで評価結果を取得

#### EX-07評価プロンプト

```
あなたは教育評価の専門家です。以下の「複合プロンプト設計」課題を評価してください。

## 評価対象
- 複合プロンプト（Phase 1 + Phase 2）
- Phase 1対話ログ
- Phase 2対話ログ
- 最終成果物
- 振り返り

## 評価基準（各20点、合計100点）

1. **2フェーズ構造（20点）**
   - Phase 1（要件定義）とPhase 2（実行）が明確に分かれているか
   - フェーズ遷移条件（「OK」等）が設定されているか

2. **Phase 1の改善ループ（20点）**
   - まとめ→提案→質問のサイクルが実装されているか
   - 要件が対話を通じて深掘りされているか

3. **Phase 2の改善ループ（20点）**
   - 成果物生成後に改善サイクルが回っているか
   - 終了条件（「完了」等）が設定されているか

4. **成果物の品質（20点）**
   - 確定要件が成果物に反映されているか
   - 品質基準（DoD）を意識しているか

5. **振り返りの質（20点）**
   - Phase 1, Phase 2それぞれの効果を分析しているか
   - 次回への改善点を特定しているか

## 出力形式
{
  "total_score": 0-100,
  "criteria_scores": {
    "two_phase_structure": 0-20,
    "phase1_loop": 0-20,
    "phase2_loop": 0-20,
    "output_quality": 0-20,
    "reflection_quality": 0-20
  },
  "feedback": "具体的なフィードバックコメント",
  "strengths": ["良かった点1", "良かった点2"],
  "improvements": ["改善点1", "改善点2"]
}
```

## 3. 評価ルーブリック詳細

### EX-07: 複合プロンプト設計

#### 課題概要

| 項目 | 内容 |
|---|---|
| 目的 | 改善ループと要件定義→実行を組み合わせた複合プロンプトを設計・実行 |
| 課題内容 | Phase 1（要件定義）+ Phase 2（実行）の複合プロンプトを作成・実行 |
| 提出物 | 複合プロンプト、Phase 1/2対話ログ、最終成果物、振り返り |
| 目安時間 | 90分 |

#### 評価ルーブリック

| 評価項目 | 優秀（5点） | 良好（4点） | 合格（3点） | 要改善（2点） | 不足（1点） |
|---|---|---|---|---|---|
| 2フェーズ構造 | 明確な分離＋遷移条件 | 分離されている | 基本構造あり | 分離が不明確 | 1フェーズのみ |
| Phase 1改善ループ | 5回以上のイテレーション | 4回 | 3回 | 2回 | ループなし |
| Phase 2改善ループ | 3回以上の修正サイクル | 2回 | 1回 | 修正なし | ループなし |
| 成果物の品質 | 実務で即使える | 調整すれば使える | 参考になる | 実用性低い | 使えない |
| 振り返りの質 | 深い洞察と改善案 | 良い分析 | 基本的な分析 | 浅い分析 | 分析なし |

|---|---|---|---|---|---|
| 依頼の明確さ | 全要素が詳細に記載 | 主要要素が明確 | 基本要素あり | 曖昧な記載 | 依頼が不明確 |
