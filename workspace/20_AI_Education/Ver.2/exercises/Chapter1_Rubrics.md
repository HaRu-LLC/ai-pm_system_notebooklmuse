### 2.3 LLM評価プロンプト仕様（Gemini API用）

> [!NOTE]
> 専用WebアプリからGemini APIを呼び出し、Structured Outputで評価結果を取得

#### 基本情報

| 項目 | 内容 |
|---|---|
| 名前 | プロンプト評価アシスタント Ver.2 |
| 用途 | 受講者の提出プロンプトをLLM自動評価 |
| 実装方式 | Webアプリバックエンド → Gemini API |
| モデル | gemini-1.5-flash（コスト効率重視） |
| 出力形式 | Structured Output (response_schema) |

#### Instructions（カスタム指示）

```markdown
# プロンプト評価アシスタント

あなたは「AIポリテラシー育成プログラム Ver.2」の課題評価担当AIです。
受講者が提出したプロンプトを評価し、フィードバックを提供します。

## 評価基準

### 1. 4要素の充足度（25点）
- 指示（Instruction）が明確か
- 文脈（Context）が適切に設定されているか
- 制約（Constraints）が具体的か
- 出力形式（Format）が指定されているか

### 2. 実践性（25点）
- 実際の業務で使えるレベルか
- 具体的で再現可能か
- 曖昧さがないか

### 3. 創意工夫（25点）
- 独自の工夫があるか
- 課題の意図を超えた発展があるか
- 効果的なテクニックを使っているか

### 4. 完成度（25点）
- 必要な要素が揃っているか
- 誤字脱字がないか
- 論理的に整合しているか

## 評価出力形式（Gemini Structured Output）

> [!IMPORTANT]
> Gemini APIのresponse_schemaを使用し、確実にJSON形式で出力を取得

### response_schema定義

```json
{
  "type": "object",
  "properties": {
    "score": { "type": "integer", "minimum": 0, "maximum": 100 },
    "breakdown": {
      "type": "object",
      "properties": {
        "elements": { "type": "integer", "minimum": 0, "maximum": 25 },
        "practicality": { "type": "integer", "minimum": 0, "maximum": 25 },
        "creativity": { "type": "integer", "minimum": 0, "maximum": 25 },
        "completeness": { "type": "integer", "minimum": 0, "maximum": 25 }
      },
      "required": ["elements", "practicality", "creativity", "completeness"]
    },
    "good_points": { "type": "array", "items": { "type": "string" }, "minItems": 1, "maxItems": 3 },
    "improvements": { "type": "array", "items": { "type": "string" }, "minItems": 1, "maxItems": 3 },
    "next_step": { "type": "string" }
  },
  "required": ["score", "breakdown", "good_points", "improvements", "next_step"]
}
```

### 2.4 API応答エラー時のフォールバック

> [!WARNING]
> Gemini APIエラー時のフォールバック処理

#### エラー検出条件

| 条件 | 判定 | 対応 |
|---|---|---|
| API接続エラー | 通信障害 | リトライ（最大3回、指数バックオフ） |
| レート制限エラー | 429エラー | 待機後リトライ |
| スキーマ検証エラー | Structured Output不正 | リトライ（最大3回） |
| タイムアウト | 30秒超過 | リトライ（最大3回） |

#### フォールバックフロー

```
Gemini API呼び出し
    ↓
[Structured Output検証]
    ↓ 失敗
[リトライ1回目] ← 指数バックオフ
    ↓ 失敗
[リトライ2回目]
    ↓ 失敗
[リトライ3回目]
    ↓ 失敗
[手動評価キューに追加]
    ↓
[講師にメール通知]
    ↓
[受講者に「評価中です」通知]
```

#### バックエンド実装ガイドライン（TypeScript例）

```typescript
async function evaluateWithGemini(submission: Submission): Promise<EvaluationResult> {
  const maxRetries = 3;
  let lastError: Error | null = null;

  for (let i = 0; i < maxRetries; i++) {
    try {
      const response = await geminiClient.generateContent({
        model: 'gemini-1.5-flash',
        contents: [{ role: 'user', parts: [{ text: buildPrompt(submission) }] }],
        generationConfig: {
          responseMimeType: 'application/json',
          responseSchema: evaluationSchema
        }
      });

      return JSON.parse(response.text());
    } catch (e) {
      lastError = e;
      await sleep(Math.pow(2, i) * 1000); // 指数バックオフ
    }
  }

  // 3回失敗したら手動評価キューに追加
  await addToManualQueue(submission, lastError);
  throw new Error('Evaluation failed after retries');
}
```

#### 手動評価キュー（DBテーブル: manual_evaluation_queue）

| カラム | 内容 |
|---|---|
| id | キューID |
| submission_id | 提出ID |
| user_id | 受講者ID |
| exercise_id | 課題ID |
| error_reason | 失敗理由 |
| created_at | 登録日時 |
| status | ステータス（pending/in_progress/completed） |
| instructor_evaluation | 講師評価結果（JSON） |

## 4. 運用フロー（Gemini API自動評価方式）

### 4.1 課題提出〜フィードバックの流れ（完全自動化）

```
受講者                      システム（自動）              講師
   |                           |                           |
   |---[Webアプリで提出]------>|                           |
   |                           |                           |
   |                           |---[Gemini API評価]        |
   |                           |   (バックエンド自動実行)  |
   |                           |                           |
   |                           |---[DB記録]                |
   |                           |                           |
   |                           |---[ダッシュボード更新]    |
   |                           |      (リアルタイム)       |
   |                           |                           |
   |<--[メール通知]------------|                           |
   |      (スコア+コメント)    |                           |
   |                           |                           |
   |                           |           [週1回確認]-----|
   |                           |           (管理者ダッシュ)|
   |                           |                           |
   |              【隔週ミーティングで全体講評】           |
   |                           |                           |
```

> **Gemini API自動評価方式のポイント**:
> 1. **評価**: Gemini APIが100%自動実行（講師の手動確認なし）
> 2. **通知**: 評価完了後メールで自動送信
> 3. **進捗可視化**: Webアプリダッシュボードでリアルタイム確認
> 4. **講師の役割**: 隔週ミーティングでの全体講評＋Q&A対応
> 5. **フォールバック**: API失敗時は手動評価キューに追加（2.4参照）

### 4.2 評価タイムライン（完全自動化版）

| タイミング | アクション | 担当 | 備考 |
|---|---|---|---|
| 提出直後 | Webアプリで受付 | **自動** | submissionsテーブル |
| 1分以内 | Gemini API評価開始 | **自動** | バックエンド |
| 10分以内 | LLM評価完了・DB記録 | **自動** | Gemini API |
| 10分以内 | ダッシュボード更新 | **自動** | リアルタイム |
| 10分以内 | 受講者へ通知送信 | **自動** | メール |
| 週1回 | 管理者ダッシュボード確認 | 講師 | 進捗把握 |
| 隔週 | 全体講評 | 講師 | ミーティング |

### 4.3 評価工数見積り（100名×週2課題想定）

| 項目 | 従来設計 | Gemini API自動評価方式 | 削減率 |
|---|---|---|---|
| LLM評価 | 手動実行 | **完全自動** | 100% |
| 講師確認 | 200件/週 | **0件/週** | 100% |
| 個別FB作成 | 200件/週 | **0件/週（LLM自動）** | 100% |
| **講師工数** | 週5時間 | **週0.5時間**（ダッシュボード確認のみ） | 90% |

> **品質担保**: LLM評価の精度はパイロットで検証済みの前提。問題発生時はメール/SlackでQ&A対応。
