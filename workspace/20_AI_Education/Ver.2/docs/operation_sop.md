## 4. 課題評価運用（Gemini API自動評価方式）

### 4.1 評価フロー（完全自動化）

```
[提出] → [Webアプリ] → [Gemini API] → [DB記録] → [自動通知]
   │          │              │            │            │
   ▼          ▼              ▼            ▼            ▼
テキスト    API呼出      LLM評価      個人ダッシュ     メール
URL/ファイル          (gemini-1.5-flash)  ボード更新
```

> **運用方針**: 品質評価はGemini APIによるLLM評価に100%委任。講師の手動確認は不要。

### 4.2 LLM評価の自動実行フロー

1. **提出受付**: Webアプリで課題を受付（テキスト/URL/ファイル）
2. **評価依頼**: バックエンドがGemini APIに提出内容を送信
3. **評価取得**: LLMがスコア（0-100）＋コメントをJSON形式で返却（10分以内）
4. **記録**: 評価結果をDB（submissions/evaluationsテーブル）に記録
5. **通知**: 受講者にメールで評価完了通知（個人ダッシュボードへのリンク付き）

### 4.3 講師の役割（評価以外に集中）

| 役割 | 頻度 | 内容 |
|---|---|---|
| 管理者ダッシュボード確認 | 週1回 | Webアプリで全体進捗・スコア分布を把握 |
| 全体講評 | 隔週 | ミーティングで共通課題・優秀例を共有 |
| Q&A対応 | 随時 | メール/Slackでの質問対応（LLM評価への疑問含む） |
| 異常値対応 | 必要時 | スコア極端に低い場合の個別フォロー |

### 4.4 全体講評の作成

**隔週ミーティングで共有**:

1. **管理者ダッシュボード共有**: Webアプリの進捗率・スコア分布をスクリーン投影
2. **優秀例の紹介**: 高スコア者のプロンプト（匿名）
3. **共通の改善ポイント**: LLMコメントから頻出課題を抽出
4. **次回への期待・ヒント**

### 4.5 評価の例外運用

| ケース | 対応 | 担当 |
|---|---|---|
| Gemini API障害時 | 手動評価に切り替え（管理者ダッシュボードから入力） | AI-PgM |
| スコア極端に低い（30点未満） | 講師が個別確認・フォローアップ | 講師 |

## 6. 教材作成ガイドライン

### 6.1 テンプレート作成ルール

> [!IMPORTANT]
> テンプレートを作成・更新する際は、以下のルールを必ず遵守すること。

**必須項目**:
| 項目 | 必須 | 説明 |
|---|:---:|---|
| プロンプト構造 | ✅ | 【指示】【文脈】【制約】【出力形式】等の枠組み |
| サンプル入力データ | ✅ | プロンプトを実行するための具体的なサンプル文章（会議メモ、レポート本文等） |
| 提出フォーマット | ✅ | 受講者が何を提出すべきかの明確な指示 |

**禁止項目**:
| 項目 | 禁止 | 理由 |
|---|:---:|---|
| AI応答結果（模範解答） | ❌ | 受講者が自分で実行した結果を見るべき。模範解答があると「正解」を模倣するだけになる |
| 講師の評価コメント例 | ❌ | 評価はGemini APIが自動実行。講師の主観的コメント例は不要 |

**設計原則**:
1. **実行可能性**: テンプレートをダウンロードしてすぐにChatGPTで実行できる状態にする
2. **再利用性**: 〇〇で置き換え可能な変数部分と、サンプルデータを明確に分離
3. **比較学習**: EX-03のように「同じ入力・異なる設定」で出力の違いを比較する課題では、共通のサンプルデータを提供

### 6.2 スライド作成ルール

| 項目 | ルール |
|---|---|
| フォーマット | Marp形式（YAML frontmatter＋Markdownスライド） |
| 構成 | 2パート制（理論編＋実践編、各約20分） |
| 学習目標 | 各動画冒頭で「この動画を終えると〇〇ができる」形式で3点明示 |
| 課題説明 | 各動画末尾に該当課題（必須/任意）の概要と取り組み方を記載 |

### 6.3 動画作成ルール

| 項目 | ルール |
|---|---|
| 時間 | 各パート約20分（合計約40分/セッション） |
| 収録形式 | スライド画面共有＋講師音声（顔出しなし） |
| 公開形式 | YouTube/Vimeo限定公開 |

## 7. 視聴ログ管理（助成金対応）

### 7.1 視聴チェックイン運用

> [!IMPORTANT]
> 助成金申請には視聴証跡が必須。Webアプリ内チェックイン方式で取得。

**チェックイン方式**:
- 各セッション動画視聴後、受講者がWebアプリで「視聴完了」ボタンを押下
- チェックインがDB（viewing_logsテーブル）に自動記録（タイムスタンプ・IPアドレス含む）
- 未チェックイン者には週次リマインド（メール自動送信）

**視聴完了判定**:
- セッション単位でチェックイン1回で「視聴完了」
- 個人ダッシュボードで視聴状況を可視化

### 7.2 視聴チェックインUI（Webアプリ）

| 要素 | 内容 |
|---|---|
| 表示場所 | 各セッション詳細ページ（動画埋め込みの下） |
| ボタン | 「視聴完了」ボタン（チェックイン済みの場合は無効化） |
| 確認 | 「動画を最後まで視聴しましたか？」確認ダイアログ |
| 記録項目 | user_id, session_id, timestamp, ip_address |

### 7.3 未チェックインリマインド

**タイミング**: 毎週金曜日（Webアプリ自動実行）

**対象**: 該当週のセッション視聴チェックインが未完了の受講者

## 12. パイロットテスト計画（Gemini API自動評価検証）

### 12.1 目的

開講前に「Webアプリ提出→Gemini API自動評価→自動通知」の完全自動化フローを検証し、LLM評価品質を確認する

### 12.2 実施概要

| 項目 | 内容 |
|---|---|
| 対象 | 社内メンバー3-5名（Type A/B/C各1名以上） |
| 期間 | 開講2週間前〜開講日 |
| 範囲 | Session 1-2のフル体験 |
| 検証項目 | **Gemini API評価品質**、Webアプリ一連フロー、ダッシュボード表示 |

### 12.3 検証項目

| 検証項目 | 成功基準 | 備考 |
|---|---|---|
| LLM評価と講師評価の一致率 | 80%以上（±10点以内） | 全任せ可否判断の基準 |
| 提出〜通知の所要時間 | 10分以内 | エンドツーエンド |
| ダッシュボード自動更新 | リアルタイム反映 | 提出後即時 |
| 受講者のUX満足度 | 4.0/5.0以上 | 特にフィードバック品質 |
| システム連携の安定性 | エラー発生率5%未満 | Webアプリ/Gemini API |
| RBACアクセス制御 | 他受講者データ閲覧不可 | セキュリティ検証 |

### 12.4 LLM評価品質の検証方法

1. パイロット参加者の提出物に対し、Gemini APIと講師が独立して評価
2. スコア差異を分析（±10点以内を「一致」とみなす）
