## 4. 観点別評価（詳細）

### 4.1 整合性（ドキュメント間の矛盾）

**整合が取れている点**
- 憲章（学習工数/KPI）→ カリキュラム（修了要件/必須任意）→ Chapter1課題（フォーム分岐）まで、学習設計の筋が揃った。
- GPTs前提が、評価運用では「OpenAI API（LLM評価）」へ置換され、技術的な実装可能性が上がった。

**残っている不整合（要修正）**
- `docs/requirements.md` の **FR-01（週2時間、4-5本）**が、憲章/カリキュラム（週90分、4本）と不一致。
- `docs/requirements.md` の **BR-03（実習提出率90%）**が、憲章/カリキュラムの「必須課題提出率100%」と不一致。
- `docs/operation_sop.md` のRACIに **「GPTs評価実行」表記**や、助成金保管先に **Notionが残存**し、Notion不使用方針と不一致。
- `docs/video_production_roadmap.md` に **Notion前提の記述**が残存（Google Sites方針と不一致）。
- `docs/test_plan.md` が旧版のままで、**Google Sites/Forms/Spreadsheet/GAS/OpenAI API**に対応したテスト観点へ更新されていない。

### 4.2 網羅性（抜け漏れ）

**揃っている**
- 研修サービスとして必要な「憲章→要件→カリキュラム→運用（SOP）→課題評価設計（ルーブリック＋自動化）」まで一通り揃った。
- 助成金の記録/証跡の枠組み、受講者の入力禁止情報/匿名化の考え方も含まれている。

**不足（追加が必要）**
- **アクセス制御（最重要）**: Spreadsheetダッシュボードを受講者に共有する場合、他受講者の氏名/メール/提出内容が閲覧できない設計が必須（行単位/個人別ビュー/権限制御の方式決定が必要）。
- **監査・ログ・アラート**: API失敗・通知失敗・コスト超過の検知とエスカレーション（誰に、何分以内に、どう通知するか）。
- **受講者向け規約**: 自動評価の性質（誤差/再評価/異議申立て）と、修了判定におけるスコアの扱い（使う/使わない）を明文化。

### 4.3 実現性（運用可能性）

**強い点**
- GASでのAPI呼び出し、フォーム連携、スプレッドシート集計、メール送信は実装現実性が高い。
- コスト上限・リトライ・障害時の手動切替の方針が要件に入ったため、運用の回復力が上がった。

**主要リスク**
- **権限設計が未確定**のままダッシュボード共有すると、個人情報/成果物が漏えいする可能性がある（設計上の最優先リスク）。
- **100%自動評価**は、受講者満足度に直結するため、パイロットでの校正（採点ブレ/モデル更新時の再校正/再評価フロー）を前提にした運用が必要。

### 4.4 ボリューム（多すぎ/少なすぎ）

**適切**
- 標準150分（必須のみで修了）＋任意/Pro Tipsで上級者にも伸び代があり、Type別の負荷設計が適切。

**調整点**
- 要件側のFR-01が旧表現のままのため、工数説明の一貫性を担保するには更新が必要。

## 5. 改善提案（具体・優先度付き）

### P0（開講前に必須）

1) **アクセス制御の設計確定（最優先）**
- 推奨案（例）:
  - 受講者は「自分の結果だけ見える」個別レポート（個人別Spreadsheet/Looker Studio/Apps Script Web App）を配布
  - 管理者のみ全体ダッシュボードを閲覧
- 必須要件: 受講者が他受講者の氏名/メール/提出物/スコアを閲覧できないこと

2) **整合性の残差を一掃（文書更新）**
- `docs/requirements.md`: FR-01の週次表現、BR-03のKPI定義を憲章/カリキュラムに合わせて更新
- `docs/operation_sop.md`: RACIの「GPTs」表記、助成金保管先のNotion表記をGoogle Appsへ統一
- `docs/video_production_roadmap.md`: Notion前提の記述をGoogle Sitesへ統一
- `docs/test_plan.md`: Google Apps＋OpenAI API前提のテスト観点を追加

3) **自動評価の品質保証（パイロットで合否判定）**
- 校正データ（サンプル提出物）を固定し、モデル更新時も同一データで再評価してブレを監視
- 異議申立て時の再評価ルール（担当/期限/反映方法）を受講者向けにも明文化

### P1（品質・信頼性を上げる）

4) **LLM評価プロンプトを“JSON専用”に統一**
- `exercises/Chapter1_Rubrics.md` の「Markdown出力」と「JSON出力」の二重仕様を整理し、API実装ではJSONのみを厳格に返す設計にする（パース失敗を防止）。

5) **助成金の証跡生成をGoogle Appsで完結**
- 視聴ログの取得方法（Vimeo/YouTubeで個人ログが取れるか）を確認し、取れない場合の代替（チェックイン、クイズ、フォーム）を設計。

## 6. 根拠（PMBOK第7版）

- 原則「**品質**」: 自動評価・自動通知はUXの中核であり、例外運用とアクセス制御まで含めて品質設計が必要。
- 原則「**リスク**」: 個人情報漏えいと誤採点は事業リスクが大きい（回避策を先に確定する）。
- 原則「**価値**」: 必須/任意・Pro TipsでType別価値を出し、完走率と満足度を両立。
- ドメイン「**デリバリー**」: 教材だけでなく運用（SOP/自動化/最終発表スケール）まで提供可能な状態に近づいた。
- ドメイン「**測定**」: KPIを取得する設計（提出・進捗・定着）が具体化された。
