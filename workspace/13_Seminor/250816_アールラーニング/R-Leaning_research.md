# 0. TL;DR（要点） {#tl-dr}
- **独自性**：中堅・中小企業の「ITかかりつけ医」＝小チーム常駐×受託のハイブリッド。  
- **強み**：教育DNA（未経験の育成、OJT、研修）×現場密着の課題解決力。  
- **生成AI**：教材開発・研修運営・社内ナレッジに大きな改善余地。導入は**リテラシー／ガバナンス**が鍵。  
- **キャリア**：エンジニア→PMOは「役割・評価・教育」を段階化。**小さな成功体験**と**制度化**が推進剤。

---

# 1. 企業概要（Snapshot） {#overview}
> 目的：事業の骨格・独自性・顧客像を短時間で把握。

| 項目 | 要旨 |
|---|---|
| 社名 | 株式会社アールラーニング（R-Learning） |
| 起源/理念 | Real-Learning（本物の学び）。「エンジニアに夢を、人と企業にHappyを」 |
| ビジネスモデル | ①**ITお助け隊**（小チーム常駐・伴走） ②受託開発（Web/アプリ/クラウド/運用） |
| 提供価値 | IT戦略相談～要件定義～開発～運用まで一気通貫。**IT部門の外部化**に相当 |
| 顧客層 | IT部門を持たない**中堅・中小企業**中心。大手・ネット企業とも取引あり |
| 組織文化 | 教育重視／OJT／質問しやすい風土。若手～ベテランの混成 |
| 差別化 | 「何でも相談できる小隊」を常備。**手薄な市場の痛点**（IT不得手）を面で解決 |
| 規模感（参考） | 中小IT企業レンジ。過去情報に**売上4億円／社員50名弱（2021頃）**の記述あり |

**ポイント**
- **小隊編成（3–4名）**で顧客現場と密着。マネジメント／インフラ／アプリを横断対応。  
- **教育×実務**の二層構造で品質を担保。育成が差別化の根。

# 2. 生成AI活用に対する課題と論点 {#genai}
> 目的：活用余地と阻害要因を可視化し、優先度を定める。

## 2.1 活用機会（ユースケース） {#genai-opportunities}
- **教材・マニュアル最適化**：難解箇所の平易化、FAQ自動生成、クイズ自動作成。  
- **研修の反転学習**：事前の自習をAIが支援。対面は応用・討議に集中。  
- **講師アシスト**：受講者の理解度に応じた**補助教材の即時生成**。  
- **24/7学習Q&A**：新人・未経験者の**質問一次受け**をAIが担い、講師負荷を軽減。  
- **社内ナレッジRAG**：お助け隊の現場知見を安全に蓄積・検索・再利用。  

## 2.2 リスク／課題（整理表） {#genai-risks}
| 区分 | 具体リスク | 失敗パターン | 推奨対策 |
|---|---|---|---|
| リテラシー | プロンプト・限界の理解不足 | 「使いどころ」が曖昧で形骸化 | **AI基礎研修**（役割別）＋社内プロンプトギャラリー |
| 品質 | 幻覚・最新性不足 | 誤情報の学習・拡散 | **人手レビュー**、RAGによる社内根拠付け |
| 著作権/IP | 学習由来の類似生成 | 無自覚な権利侵害 | **利用ガイドライン**、引用・出典の明示プロセス |
| セキュリティ | 機微情報の外部投入 | データ漏えい | **匿名化／自社ホスト／権限管理**の原則 |
| 教材設計 | AI任せで思考退行 | 学習効果が不安定 | **人×AIハイブリッド設計**（設問設計・振り返り強化） |

## 2.3 小さく賢く始める（実行シーケンス） {#genai-sequence}
1) **サンドボックス環境**（社内閉域 or ガバナンス設定済みSaaS）。  
2) **パイロット**（1チーム×2ヶ月）：教材平易化＋FAQ自動生成のKPI検証。  
3) **RAG最小構成**：お助け隊の標準手順書・トラブル事例をインデックス化。  
4) **運用ルール**：情報分類、プロンプトテンプレ、レビュー責任者の明確化。  
5) **段階拡大**：対象研修・顧客支援へ水平展開。**定着指標**でモニタリング。

**KPI例**：教材作成時間▲30%、新人立ち上がり期間▲25%、講師質問対応時間▲40%、FAQ参照率↑。

# 3. エンジニア → PMO キャリアプランと課題 {#pmo}
> 目的：役割・スキル・教育・評価を一体設計し、移行を現実路線で。

## 3.1 PMOに必要な力（スキルマップ） {#pmo-skill}
| スキル領域 | 中核スキル | 具体行動例 |
|---|---|---|
| PM基礎 | WBS/進捗/リスク/課題管理 | 週次レビュー、早期エスカレーション |
| 調整力 | ファシリ・交渉・合意形成 | 論点整理→選択肢→意思決定の議事運営 |
| ビジネス理解 | 業務/ドメイン/ROI | 要件→業務効果への翻訳、優先度付け |
| コミュニケーション | ステークホルダー連携 | 伝達の型（目的→結論→根拠→依頼） |
| IT知識 | 開発工程・品質・インフラ | 技術課題の深刻度評価、優先順位付け |
| ドキュメント | 議事録・計画・報告書 | 「決定・宿題・期日」の明示と追跡 |

## 3.2 学習設計（6–9ヶ月モデル） {#pmo-curriculum}
- **Phase0（1ヶ月）**：PMBOK/スクラム速習、テンプレ群（議事・課題台帳・進捗レポート）。  
- **Phase1（2–3ヶ月）**：小規模案件で**PMO補佐**（議事録/イシュー整理/タスク追跡）。  
- **Phase2（2–3ヶ月）**：中規模で**PMO主担当**（変更管理/リスク審査/定例運営）。  
- **Phase3（1–2ヶ月）**：**横断PMO**（複数案件の資源配分・標準プロセス適用）。  
- 併走：**生成AI活用**（議事要約、KPT集約、レポート自動草案、RAG参照）。

**評価KPI**：遅延検知の早期化、未然防止件数、定例会の決定率、文書の再利用率、満足度。

## 3.3 制度・運用（移行の現実解） {#pmo-system}
- **役割の段階化**：SE→**PMO補佐**→PMO→PM。職務定義と昇格要件を明文化。  
- **小さな成功体験**：まずは**1チームで定例運営の可視化**（OKR/ダッシュボード）。  
- **ジョブローテ**：要件・品質・調達・変更管理を**回遊**し、抜け漏れ防止。  
- **メンタリング**：先輩PM/PMOの**シャドーイング**→リバースメンタリングで定着。  
- **評価の見える化**：プロジェクト成功率、標準化貢献、顧客/PMの**360°FB**を反映。

## 3.4 リスクと対処 {#pmo-risks}
| リスク | 症状 | 兆候指標 | 予防・対処 |
|---|---|---|---|
| 技術リソース不足 | 兼務で破綻 | 残業増、欠陥増 | 代替育成・引継ぎ計画、**工数ゲート** |
| マインド転換 | 「自分でやる方が速い」 | 依頼/委譲が滞る | サーバント型教育、成功体験設計 |
| 教育機会偏在 | OJT任せ | 経験の穴 | カリキュラム×ローテ×外部研修 |
| 評価不透明 | 貢献が見えない | モチベ低下 | KPI定義、**360°評価**、昇格要件の公開 |

# 4. 講演用メッセージ（示唆） {#talk-track}
- **学びが中核の企業**は、生成AIで**学び方そのもの**を革新できる。  
- **お助け隊×RAG**で「現場知見の循環」を高速化。小隊の経験が**資産**になる。  
- エンジニアは**PMOの器**を段階的に鍛えると、**年齢が価値**に変わる職務に昇華する。  
- 小さく始め、**成果KPI→標準化→水平展開**が勝ち筋。
