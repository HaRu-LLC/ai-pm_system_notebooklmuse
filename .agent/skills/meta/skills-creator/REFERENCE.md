## 1. Skill設計の原則

### 1.1 Single Responsibility（単一責任）
- 1つのSkillは1つのことを上手にやる
- 複数の目的がある場合は分割を検討
- 「このSkillは〇〇するもの」と一言で説明できること

### 1.2 User-Centric（ユーザー中心）
- 技術的に正しいよりも、ユーザーが使いやすいを優先
- 専門用語は避け、わかりやすい言葉で
- 入力は最小限、出力は最大限

### 1.3 Fail-Safe（安全設計）
- 不完全な入力でも動作する（デフォルト値）
- エラー時の対処方法を明示
- 仮説は仮説と明記する

## 2. トリガー語の設計ガイド

### 良いトリガー語の特徴
- **自然**: 普段の会話で使う言葉
- **具体的**: 他のSkillと区別できる
- **記憶しやすい**: 長すぎない（2〜5単語）

### トリガー語のパターン

| パターン | 例 |
|----------|-----|
| 動詞 + 目的語 | 「提案書を作って」「リスクを分析」 |
| 〇〇したい | 「テンプレを作りたい」 |
| 〇〇のレビュー | 「セキュリティのレビュー」 |
| Skill名そのもの | 「Skills Creator」 |

### 避けるべきトリガー語
- ❌ 汎用的すぎる（「ドキュメント作成」→ どのドキュメント？）
- ❌ 他Skillと重複（registry.mdで確認）
- ❌ 技術用語のみ（「SKILL.md生成」→ 一般ユーザーは使わない）

## 3. 入力設計のベストプラクティス

### 質問数の目安
- **必須**: 3項目（これがないと動かない）
- **推奨**: 5項目以内（多すぎると面倒）
- **任意**: 必要に応じて（デフォルト動作を用意）

### 質問の構造
```
<質問文>（<なぜこの情報が必要か>）
```

例:
- 「参考資料はありますか？（型を踏襲するため）」
- 「誰が読みますか？（トーンを調整するため）」

### 入力がない場合の対処
1. **デフォルト値を使用**: 最も一般的なケースを想定
2. **追加質問で補完**: 手順の中で確認
3. **仮説として進む**: 明示した上で進行

---

## 4. 手順設計のベストプラクティス

### ステップ数の目安
- **最小**: 3ステップ（シンプルなSkill）
- **標準**: 5ステップ（一般的なSkill）
- **最大**: 7ステップ（複雑なSkill、これ以上は分割を検討）

### ステップの粒度
- 1ステップ = 1つの明確なアクション
- 「〇〇を確認し、△△を作成」は2ステップに分割
- 各ステップに成果物または状態変化がある

### 分岐の扱い
```markdown
### Step 3: 参照資料の確認
- **参照資料がある場合**: Step 4へ
- **参照資料がない場合**: 汎用テンプレートを使用してStep 4へ
```

## 5. 成果物設計のベストプラクティス

### 成果物の種類

| 種類 | 説明 | 例 |
|------|------|-----|
| 主成果物 | 必ず生成される | 提案書.md |
| 副成果物 | オプションで生成 | チェックリスト.md |
| 中間成果物 | 手順途中で生成 | 分析メモ.md |

### 命名規則
```
<種別>_<対象>_<日付>.md
```
例: `PROPOSAL_AI導入_20251228.md`

### 保存先の決定
- **汎用**: `.agent/templates/` または `.agent/outputs/`
- **プロジェクト固有**: `workspace/{ProjectName}/docs/`
- **一時的**: ユーザーに確認してから決定

## 6. 検証設計のベストプラクティス

### 良い完了条件の特徴
- **客観的**: 誰が見ても同じ判定
- **測定可能**: 数値または存在確認
- **実行可能**: チェック方法が明確

### 完了条件のパターン

| パターン | 例 |
|----------|-----|
| 存在確認 | 「〇〇ファイルが保存されている」 |
| 数量確認 | 「3つ以上の選択肢が提示されている」 |
| 品質確認 | 「仮説部分が【仮説】とマークされている」 |
| 整合性確認 | 「参考資料と同等の構成になっている」 |

### 避けるべき完了条件
- ❌ 「良い感じになった」（主観的）
- ❌ 「ユーザーが満足した」（測定不能）
- ❌ 「完璧になった」（完璧の定義がない）

## 7. カテゴリ判定フローチャート

```
[Start]
   │
   ├─ ビジネス文書・経営判断に関係？ ─Yes→ business/
   │
   ├─ プロジェクト管理・リスクに関係？ ─Yes→ pm/
   │
   ├─ 開発・実装・技術設計に関係？ ─Yes→ engineering/
   │
   └─ Skill生成・自動化基盤に関係？ ─Yes→ meta/
```

### 迷った場合
1. **主な利用者**で判断（PM向け → pm/、エンジニア向け → engineering/）
2. **成果物の種類**で判断（ビジネス文書 → business/、コード → engineering/）
3. **複数該当**する場合は、最も頻繁に使うユーザーのカテゴリ

## 8. 品質チェックリスト

Skill生成後、以下を確認：

### 構造チェック
- [ ] frontmatter（name, description）が正しい
- [ ] 全セクション（目的、トリガー、入力、手順、成果物、検証）がある
- [ ] Markdown構文が正しい

### 内容チェック
- [ ] 目的が2-3文で明確に説明されている
- [ ] トリガー語が3つ以上ある
- [ ] 入力質問に「なぜ必要か」が含まれている
- [ ] 手順が3-7ステップに収まっている
- [ ] 各ステップが具体的で実行可能
- [ ] 成果物が明確に定義されている
- [ ] 完了条件が客観的に判定可能

### 整合性チェック
- [ ] 既存Skillとトリガー語が重複していない
- [ ] registry.mdに登録されている
- [ ] 関連Skill/Workflowへの参照が正しい
