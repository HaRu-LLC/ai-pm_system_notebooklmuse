## 1. 要件分析時のパターン

### パターン1-1: 要求が曖昧
**状況**: 「便利なSkillが欲しい」のような抽象的な要求

**対処**: 具体例を3つ挙げてもらう
```
「例えば、どんな場面で使いたいですか？3つほど教えてください」
```

**次のアクション**: 具体例から共通点を抽出してゴールを定義

---

### パターン1-2: スコープが広すぎる
**状況**: 「プロジェクト管理を全部やるSkill」のような大きすぎる要求

**対処**: 最小限の機能から始める
```
「まず最も重要な機能だけに絞ると、何ですか？」
```

**次のアクション**: コア機能のみでV1を作成、拡張はV2以降

---

### パターン1-3: 既存Skillと重複
**状況**: 既存Skillとほぼ同じ機能を要求

**対処**: 差分を明確にする
```
「既存の〇〇Skillとの違いは何ですか？」
```

**次のアクション**:
- 差分が小さい → 既存Skillの拡張を提案
- 差分が大きい → 新規Skillとして作成

---

### パターン1-4: 技術的に実現困難
**状況**: AIの限界を超える要求（例：完璧な予測）

**対処**: 代替案を提示
```
「〇〇は難しいですが、△△なら実現可能です。いかがですか？」
```

**次のアクション**: 実現可能な範囲で価値を最大化

## 2. 設計時のパターン

### パターン2-1: 入力が多すぎる
**状況**: 必須入力が10個以上になりそう

**対処**:
1. 本当に必須なのは3つだけに絞る
2. 残りはデフォルト値を設定
3. 高度な設定は「任意」セクションへ

---

### パターン2-2: 手順が複雑すぎる
**状況**: ステップが10個以上になる

**対処**:
1. フェーズに分ける（Phase 1〜3程度）
2. 各フェーズ内を3ステップ以下に
3. または、Skillを分割する

---

### パターン2-3: 成果物が不明確
**状況**: 「いい感じのドキュメント」のような曖昧な成果物

**対処**:
1. 具体的なファイル形式を決める（.md / .json / .csv）
2. サンプルを見せて確認
3. 最低限含める要素をリスト化

---

## 3. 生成時のパターン

### パターン3-1: SKILL.mdが500行を超える
**状況**: 詳細を書きすぎて500行超過

**対処**:
1. 詳細説明はREFERENCE.mdへ移動
2. 「→ 詳細は[REFERENCE.md]参照」でリンク
3. 本体には要点のみ残す

---

### パターン3-2: トリガー語が決まらない
**状況**: 良いトリガー語が思いつかない

**対処**:
1. ユーザーが実際に言いそうな言葉を想像
2. 「〇〇したい」「〇〇を作って」形式で考える
3. 既存Skillと重複しないか確認

---

### パターン3-3: 検証条件が主観的
**状況**: 「良い品質になった」のような主観的条件

**対処**:
1. 「〇〇ファイルが存在する」のような客観的条件に
2. 「〇〇が3つ以上含まれる」のような数値条件に
3. チェックリスト形式にする

---

## 4. 統合時のパターン

### パターン4-1: カテゴリが決まらない
**状況**: business/とpm/のどちらにも該当しそう

**対処**:
1. **主な利用者**で判断（PMが使う → pm/）
2. **成果物の種類**で判断（ビジネス文書 → business/）
3. 迷ったらより具体的なカテゴリを選ぶ

---

### パターン4-2: 既存Skillとの連携
**状況**: 他のSkillと組み合わせて使うことが多そう

**対処**:
1. 「参照」セクションに関連Skillを記載
2. 呼び出しチェーンを設計（A → B → C）
3. 入力/出力の形式を揃える

---

## 5. 使用例テンプレート

### 例1: ドキュメント生成系Skill

```
ユーザー: 「議事録を自動生成するSkillが欲しい」

Skills Creator:
1. 要件分析 → カテゴリ: pm/、成果物: 議事録.md
2. 設計:
   - 入力: 会議メモ、参加者、日付
   - 手順: 構造化 → 整形 → 保存
   - 成果物: 議事録Markdown
3. 生成 → meeting-minutes-generator Skill作成
4. 統合 → registry.md更新
```

---

### 例2: 分析系Skill

```
ユーザー: 「競合分析をしたい」

Skills Creator:
1. 要件分析 → カテゴリ: business/、成果物: 分析レポート
2. 設計:
   - 入力: 自社情報、競合リスト、分析観点
   - 手順: 情報収集 → 比較分析 → レポート生成
   - 成果物: 競合分析レポート.md
3. 生成 → competitor-analysis Skill作成
4. 統合 → registry.md更新
```

---

### 例3: レビュー系Skill

```
ユーザー: 「コードレビューを効率化したい」

Skills Creator:
1. 要件分析 → カテゴリ: engineering/、成果物: レビューコメント
2. 設計:
   - 入力: コード/PR、レビュー観点、重要度
   - 手順: コード読解 → 問題検出 → コメント生成
   - 成果物: レビューレポート.md
3. 生成 → code-review-assistant Skill作成
4. 統合 → registry.md更新
```
