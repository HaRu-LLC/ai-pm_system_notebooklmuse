# メール取得機能のプライバシーレビュー

メール（PII）取得で起きがちな事故を事前に防ぐ設計レビューワークフロー。

---

## 入力

1. **メール取得の目的**（結果送付/ログイン/連絡/マーケ等）
2. **保存先**（DB/外部サービス/ローカル）
3. **保持期間**（未定なら仮案）
4. **削除要件**（ユーザーからの削除依頼への対応）

---

## レビュー項目チェックリスト

### 1. 目的の正当化
- [ ] メール取得が本当に必要か？
- [ ] 代替手段（匿名ID、ワンタイムコード等）で代替できないか？
- [ ] 目的が明確で、オーバーコレクションになっていないか？

### 2. 同意（オプトイン）
- [ ] 同意文言はユーザーに理解しやすいか？
- [ ] プライバシーポリシー/通知へのリンクがあるか？
- [ ] チェックボックス等で明示的同意を取得しているか？
- [ ] 二重同意（ダブルオプトイン）が必要なケースか？

### 3. 取得最小化
- [ ] フォーム項目は最小限か？（メール以外に不要な項目がないか）
- [ ] オプション項目と必須項目は区別されているか？

### 4. 露出経路遮断
- [ ] URLクエリパラメータにメールを載せていないか？
- [ ] ログにメールがマスクなしで出力されていないか？
- [ ] エラー画面・通知にメールが露出していないか？
- [ ] 分析用ツールにメールがそのまま送信されていないか？

### 5. 保存と保護
- [ ] メールは分離保管されているか？（他テーブル/他DBと分離）
- [ ] 暗号化（at rest / in transit）されているか？
- [ ] アクセス制御（誰がアクセスできるか）が明確か？

### 6. 保持と削除
- [ ] 保持期間が決まっているか？
- [ ] 削除手順が整備されているか？
- [ ] ユーザーからの削除依頼への導線があるか？

### 7. インシデント対応
- [ ] 漏えい時の対応手順が（最小限でも）記載されているか？
- [ ] 報告先・連絡先が明確か？
