# Cursor Command: 受け入れ基準書

## 説明
サブプロジェクトの**受け入れ基準書**を生成します。
機能/非機能/統合の受け入れ基準を明確化し、受け入れテスト（AT/UAT）の判定基準を定義します。
受け入れ基準作成自体をプロジェクトタスクとして管理します。

---

## プロンプトテンプレート

あなたはPMBOK第7版準拠の全社横断AIプロジェクトマネージャー（AI-PgM）です。
以下の情報をもとに、**受け入れ基準書**をMarkdown形式で作成してください。

### 入力変数
- `target_subpj`: 対象サブプロジェクト名
- `version_date`: 版/日付
- `approver`: 承認者
- `functional_criteria`: 機能受け入れ基準（機能ID、機能名、受け入れ条件、測定方法、合格基準）
- `non_functional_criteria`: 非機能受け入れ基準（項目、基準、測定方法、合格基準）
- `integration_criteria`: 統合受け入れ基準（連携パターン、受け入れ条件、測定方法、合格基準）
- `test_period`: 実施期間
- `test_organization`: 実施体制（役割）
- `test_env`: テスト環境
- `judgment_date`: 判定会議日

---

## 出力テンプレート

```markdown
# 受け入れ基準書
- 対象サブPJ: {{target_subpj}}
- 版/日付: {{version_date}}
- 承認者: {{approver}}

## 1. 機能受け入れ基準
| 機能ID | 機能名 | 受け入れ条件 | 測定方法 | 合格基準 |
|---|---|---|---|---|
{{functional_criteria}}

## 2. 非機能受け入れ基準
| 項目 | 基準 | 測定方法 | 合格基準 |
|---|---|---|---|
{{non_functional_criteria}}

## 3. 統合受け入れ基準（サブPJ間連携）
| 連携パターン | 受け入れ条件 | 測定方法 | 合格基準 |
|---|---|---|---|
{{integration_criteria}}

## 4. 受け入れテスト実施計画
- 実施期間: {{test_period}}
- 実施体制: {{test_organization}}
- 環境: {{test_env}}
- 判定会議: {{judgment_date}}

## 5. 受け入れ判定基準
- **合格**: すべての受け入れ基準を満たす
- **条件付き合格**: 軽微な不具合あり、期限内に修正コミット
- **不合格**: 重大な不具合あり、再テスト必要
```

## 使用例

```
/受け入れ基準書を作成

target_subpj: "LLM構築（楊氏）"
version_date: "v1.0 / 2025-03-01"
approver: "王PM"
functional_criteria: |
  | F-01 | チャット応答機能 | 質問に対して適切な回答を返す | シナリオテスト | 正答率90%以上 |
  | F-02 | RAG検索機能 | 社内ドキュメントから関連情報を取得 | 検索精度測定 | Recall@10 > 80% |
  | F-03 | ファインチューニング | 専門用語を正しく理解 | ドメイン特化テスト | 専門用語正答率85%以上 |
non_functional_criteria: |
  | 性能 | レスポンスタイム | 負荷テスト | 95%ile < 3秒 |
  | 可用性 | 稼働率 | 監視ログ分析 | >99.5% |
  | セキュリティ | 情報漏洩防止 | セキュリティスキャン | 重大な脆弱性0件 |
  | スケーラビリティ | 同時接続数 | 負荷テスト | 100ユーザー同時接続可能 |
integration_criteria: |
  | LLM → 流体解析エージェント | APIレスポンス正常 | 統合テスト | エラー率<0.1% |
  | LLM → 暗黙知エージェント | データ連携正常 | 統合テスト | データ整合性100% |
test_period: "2025-03-25 ～ 2025-04-10"
test_organization: "テスト責任者: 王PM / 実施者: 品質チーム"
test_env: "ステージング環境（TerraData）"
judgment_date: "2025-04-15"
```

## 備考
- **原則**: 品質、価値
- **ドメイン**: デリバリー、測定
- VT（ベンダー側テスト）完了・承認済みが前提条件
- 受け入れ基準書承認が受け入れテスト計画立案の前提
